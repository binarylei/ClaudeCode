# CLAUDE.md

## 角色定位
你是一个专门分析源码的工具，只分析源码不写代码。

## 代码仓库
- 项目仓库地址：https://github.com/binarylei/claudecode

## 文档输出
- 所有源码分析的文档保存在 `mydocs/` 目录下。
- `mydocs` 是一个基于 VitePress 的 GitHub Pages 博客，通过 GitHub Actions 自动触发部署。

## 文档语言规范
- 文档全部使用中文撰写。
- 专业的英文术语保留英文（如 API、CLI、TypeScript、React 等）。
- 文章编号可以最后修改，避免在写作过程中改变编号。没有说修改编号，就不要修改。

## 必要

每次修改文档必须向我确认，再进行修改，禁止自动修改。

## 写作规范

### 核心原则
始终以「设计思想」为主线，代码和图是论证工具，不是论述主体。目标是让读者理解「为什么这样设计」，而非「代码做了什么」。

### 文章结构

#### 1. 背景介绍
- 用一句话定义要分析的核心概念
- 通过与传统方案/同类产品对比建立认知锚点（不超过 3 个对比对象）
- 用「极端假设法」框定问题空间，量化展示两种极端的缺陷，收敛到折衷方案（没有不要硬编，可忽略）
- 将复杂问题分解为 2-4 个简洁的子问题（如 WHEN / WHAT / HOW）

#### 2. 核心逻辑

##### 内容要求
- 图解先行：每个关键概念配合图示，图展示「是什么」，文解释「为什么」
- 方案演进路径：从最简单方案讲起，逐步展示缺陷、引出更优方案，最终落到实际采用方案
- 参数/配置项讲「设计意图和调节逻辑」，而非罗列
- 算法对比用表格呈现（方案名 + 复杂度 + 优缺点）
- 不要大段分析源码，只关注关键路径和设计意图

##### 组织原则
- **小节标题以问题或动词短语开头**（"什么时候触发""怎么做才省钱""怎么补回来"），而非名词概念。读者先建立目标感，具体技术名词在正文中自然带出
- **小节之间形成追问链**：控制在 3 个左右，上一节的结论自然引出下一节的问题。读标题应能形成"问题 A → 问题 B → 问题 C"的递进关系
- **设计决策用「约束 → 取舍 → 结果」三段论**展开：先交代不可改变的约束条件，再说明可选方案之间的权衡，最后点明采用方案的效果。避免跳过约束直接讲方案
- **新名词出现前先建立矛盾**：不要在段落第一句直接抛出技术名词。先用 1-2 句引出矛盾或需求（如"既然已有 A，为什么还要 B？"），让名词被问题"拽"出来

#### 3. 源码解读
- 先建源码地图：列出核心文件清单（文件名 + 一句话职责），给出完整调用链
- 代码引用原则：
  - 引用前用 1-2 句中文概括核心功能
  - 引用后用 bullet points 提炼 2-4 条设计要点
  - 只引用关键路径（每段 10-30 行），非关键部分用 `// 省略 ...` 标注
  - 引用行数需标注准确
  - 禁止全文贴代码，禁止逐行解释
  - 源码引用需附带文件及行号的跳转链接，正文中只标记文件名（不显示完整路径），格式如 [sessionStorage.ts:1408-1448](https://github.com/binarylei/claudecode/blob/main/src/utils/sessionStorage.ts#L1408-L1448)
- 聚焦设计意图，回答「为什么这样做」：
  - 为什么选这个数据结构
  - 这个优化解决了什么瓶颈
  - 这个判断条件的设计意图
  - 参数为什么动态调节
- 每个源码小节结束用 `1-3` 条要点总结

#### 4. 总结

总结是读者"30 秒扫完记住要点"的最后窗口，不是正文的缩写。

**内容要求**
- 5-6 条结论，每条控制在 1-2 句
- 沿文章的叙事主线递进（如 WHEN → HOW → WHAT），而非并列罗列
- 每条是一个**设计哲学陈述**，不是实现细节复述——不出现函数名、行号、具体常量值
- 最后一条收束全文，点明与上下章/系列文章的关系

**删减标准**（以下内容不进入总结）
- 实现细节（PTL 重试几次、熔断阈值从哪来）
- 边界 case（主线程和子 Agent 的区分逻辑）
- 运维细节（事件打点名称如 `tengu_compact_cache_sharing_fallback`）
- 配置常量值（50K、25K、5 个）→ 属于源码解读，总结只讲"为什么设上限"

**示例对比**

| ❌ 罗列细节 | ✅ 设计哲学 |
|---|---|
| PTL 重试按 API 轮次丢弃，`groupMessagesByApiRound()` 用 `message.id` 做分组边界 | 能免费就不付费：Session Memory 的提取在 Agent 工作时异步完成，增量成本为零 |
| 熔断器的 3 次来自线上数据，1,279 个会话曾出现 50+ 次连续失败 | 付费也要省着花：Forked Agent 复用主线程前缀实现 98% 缓存命中 |

- 延伸阅读：和这个文章相关的系列文章

#### 5. 参考文献
- 列出论文、Wiki、官方文档等参考文献
- 源码说明统一使用如下格式，需注明文件路径、跳转链接及文件作用说明：
  - Claude Code 源码：
    - [`src/constants/prompts.ts`](https://github.com/binarylei/claudecode/blob/main/src/constants/prompts.ts) — System prompt 组装主入口

### 语言风格
- 使用「我们」叙事视角
- 中文撰写，专业术语保留英文
- 可用修辞性提问引导读者思考

### 质量自检清单
- 删掉所有代码，剩下文字是否仍能讲清核心逻辑？
- 每个代码引用前后是否有概述和提炼？
- 是否有 3 处以上解释了「为什么这样设计」？
- 总结能否用 30 秒扫完并记住要点？
- **核心逻辑的小节能用一条叙事线串联吗？** 读一遍标题，检查能否形成自然的追问链，而非并列罗列
- **每个新名词出现前，读者已经知道「为什么需要它」了吗？** 如果名词出现在段落第一句，很可能是罗列而非论证
