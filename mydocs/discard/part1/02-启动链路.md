# 02. 启动链路

> 从敲下 `claude` 到 REPL 就绪——一条命令背后的 8 个阶段。

---

## 1. 概述

启动链路是 Claude Code 最复杂的单条执行路径。它串联了 4 个核心文件、跨越 8 个逻辑阶段，在约数百毫秒内完成从命令行参数解析到 React 终端 UI 渲染的全过程。

**核心设计原则**：

| 原则 | 含义 | 体现 |
|------|------|------|
| 懒加载 | 不到用时不动代码 | 全链路动态 `import()` |
| I/O 重叠 | 等待不浪费 CPU | 预取与模块加载并行 |
| 快速路径优先 | 不需要的就别加载 | `--version` 零导入返回 |
| 安全分阶段 | 信任前少做事 | 环境变量分两次应用 |

---

## 3. 阶段 1：开发入口 `dev-entry.ts`

### 3.1 职责

`dev-entry.ts` 是开发环境独有的启动垫片。在生产构建中，Bun 的构建工具会将其替换为直接入口；但在本仓库（source map 还原版本）中，它是实际的启动文件。

核心职责只有两个：
1. **注入构建时宏**——版本号、构建时间、包下载地址
2. **检测缺失模块**——递归扫描 `src/` 和 `vendor/`，确保所有导入可解析

### 3.2 MACRO 全局对象

```typescript
// dev-entry.ts 第 5-13 行：类型定义（7 个字段，包括版本号、包地址、反馈渠道等）
// dev-entry.ts 第 15-29 行：从 package.json 构造默认值并条件注入
const defaultMacro: MacroConfig = {
  VERSION: pkg.version,
  BUILD_TIME: '',                  // 还原版本无构建时间
  PACKAGE_URL: pkg.name,
  // ... ISSUES_EXPLAINER, FEEDBACK_CHANNEL 等
}

if (!('MACRO' in globalThis)) {
  ;(globalThis as typeof globalThis & { MACRO: MacroConfig }).MACRO = defaultMacro
}
```

**条件注入**是核心：先检查 `globalThis` 上是否已有 `MACRO`。生产构建中 Bun 在打包时直接注入这些值；还原版本从 `package.json` 读取，保证后续代码不需要感知差异。**`BUILD_TIME` 为空**——还原版本没有真实的构建时间戳。

`MACRO` 在整个代码库中作为全局常量被引用，所有依赖版本号或构建信息的模块都通过它取值而非硬编码。

### 3.3 缺失导入检测

```typescript
// dev-entry.ts 第 67-96 行
function collectMissingRelativeImports(): MissingImport[] {
  const files: string[] = []
  scanFiles(resolve('src'), files)     // 递归收集 src/ 和 vendor/ 下所有源文件
  scanFiles(resolve('vendor'), files)

  // 正则匹配 ESM import/export 和 CJS require 的相对路径
  const pattern =
    /(?:import|export)\s+[\s\S]*?from\s+['"](\.\.?\/[^'"]+)['"]|require\(\s*['"](\.\.?\/[^'"]+)['"]\s*\)/g

  // 对每个匹配项调用 hasResolvableTarget() 尝试解析，缺失则记录并去重
  // hasResolvableTarget()（第 50-65 行）对每个路径尝试 9 种后缀组合：
  //   无后缀 → .ts / .tsx / .js / .jsx / .mjs / .cjs
  //          → /index.ts / /index.tsx / /index.js
}
```

这是还原项目的**自愈机制**：source map 还原后的 `.js` 文件可能引用尚未还原的 `.ts` 源文件。缺失时 CLI 不会抛出晦涩的 `ERR_MODULE_NOT_FOUND`，而是在启动时打印清晰的缺失模块列表，引导开发者去还原对应的文件。

### 3.4 快速路径：`--version` 和 `--help`

与生产版本不同，还原版本的 `--version` 和 `--help` **仍然会运行缺失导入检测**，但不会因缺失模块而阻止输出：

```typescript
// dev-entry.ts 第 101-126 行

// --version：存在缺失模块时额外标注 "(restored dev workspace)"
if (args.includes('--version')) {
  if (missingImports.length > 0) {
    console.log(`${pkg.version} (restored dev workspace)`)
    console.log(`missing_relative_imports=${missingImports.length}`)
    process.exit(0)
  }
  console.log(pkg.version)
  process.exit(0)
}

// --help：存在缺失模块时打印还原工作区信息
if (args.includes('--help')) {
  if (missingImports.length > 0) {
    console.log('Claude Code restored development workspace')
    console.log(`version: ${pkg.version}`)
    console.log(`missing relative imports: ${missingImports.length}`)
    process.exit(0)
  }
  console.log('Usage: claude [options] [prompt]')
  process.exit(0)
}
```

**存在缺失模块时的最终行为**（第 128-141 行）——不是快速路径，而是阻止启动：

```typescript
if (missingImports.length > 0) {
  // 打印前 20 个缺失模块
  for (const item of missingImports.slice(0, 20)) {
    console.log(`- ${item.importer} -> ${item.specifier}`)
  }
  console.log('The original app entry is still blocked by missing restored sources.')
  console.log('Use this workspace to continue restoration; once missing imports reach 0, this launcher will forward to src/main.tsx automatically.')
  process.exit(0)  // ← 不加载 main.tsx，直接退出
}

// 只有缺失导入数为 0 时，才会路由到真正的 CLI 入口
await import('./entrypoints/cli.tsx')
```

这是一个**自动门禁**：缺失导入数 > 0 时，`dev-entry.ts` 就是终点——不会加载 `cli.tsx`，更不会加载 `main.tsx`。只有当所有源文件就绪后，门禁才自动放行。

## 4. 阶段 2：CLI 路由器 `entrypoints/cli.tsx`

### 4.1 职责

`cli.tsx` 是 CLI 的快速路由层。它的核心策略是**最小化模块加载**——所有特殊标志都在加载 `main.tsx`（那个 808KB 的巨型模块）之前就被处理并返回。

> **环境变量必须在模块导入前设置**：`cli.tsx` 顶级作用域（`main()` 之前）设置了消融基线等环境变量。原因在于 `BashTool`、`AgentTool` 等在**模块导入时**就将这些环境变量捕获为模块级常量——等到 `init()` 再设置已来不及。

### 4.2 快速路径路由表

`main()` 函数按优先级顺序检查命令行参数，匹配到快速路径后立即返回：

| 优先级 | 触发条件 | 操作 | 加载模块数 |
|--------|---------|------|-----------|
| 1 | `--version` / `-v` | 打印版本号后退出 | 0 |
| 2 | `--dump-system-prompt` | 加载配置 + 模型，打印系统提示 | ~3 |
| 3 | `--claude-in-chrome-mcp` | 启动 Chrome MCP 服务器 | ~5 |
| 4 | `--chrome-native-host` | 启动 Chrome 原生消息主机 | ~5 |
| 5 | `--computer-use-mcp` | 启动 Computer Use MCP 服务器 | ~5 |
| 6 | `--daemon-worker=<kind>` | 加载并运行守护进程 worker | ~3 |
| 7 | `remote-control` / `rc` / `bridge` | 认证 + policy 检查 + 桥接启动 | ~8 |
| 8 | `daemon [subcommand]` | 守护进程管理 | ~5 |
| 9 | `ps` / `logs` / `attach` / `kill` | 后台会话管理 | ~3 |
| 10 | `--bg` / `--background` | 后台任务模式 | ~5 |
| 11 | `new` / `list` / `reply` | 模板任务 | ~3 |
| 12 | `environment-runner` | 无头 BYOC 运行器 | ~5 |
| 13 | `self-hosted-runner` | 无头自托管运行器 | ~5 |
| 14 | `--tmux` + `--worktree` | 进入 tmux 工作树 | ~5 |

### 4.3 最终路径

只有在**没有匹配任何快速路径**时（即用户要进行正常的对话交互），才会进入最终路径：

```typescript
// cli.tsx 第 287-298 行
await import('../utils/earlyInput.js')   // 1. 开始缓冲提前输入
profileCheckpoint('cli_before_main_import')
await import('../main.js')               // 2. 动态加载主模块
profileCheckpoint('cli_after_main_import')
await cliMain()                          // 3. 调用 main.tsx 的导出 main()
profileCheckpoint('cli_after_main_complete')
```

三个步骤，三个 `profileCheckpoint` 性能埋点：

| 检查点 | 含义 |
|--------|------|
| `cli_before_main_import` | 即将加载主模块 |
| `cli_after_main_import` | 主模块导入完成（含~200个依赖） |
| `cli_after_main_complete` | 整个启动流程结束 |

**`startCapturingEarlyInput()`** 是一个巧妙的机制：在 REPL 就绪之前，用户可能在终端输入了内容（如粘贴文本）。这个函数会缓冲 stdin 的早期输入，等 REPL 启动后再注入——避免输入丢失。

## 5. 阶段 3：模块副作用 `main.tsx` 顶级

### 5.1 职责

`main.tsx` 的顶级代码在**被导入时**就立即执行。这一步做了两件事：
1. 启动 I/O 预取（与模块加载并行）
2. 导入 ~200 个模块（被预取的时间所覆盖）

### 5.2 启动 I/O 预取

```typescript
// main.tsx 第 12-20 行
profileCheckpoint('main_tsx_entry')

startMdmRawRead()        // 启动 MDM 策略读取（plutil / reg query）
                         // 这是一个子进程，与后续导入并行运行

startKeychainPrefetch()  // 启动 macOS 钥匙串读取
                         // 预取 OAuth token 和传统 API key
```

这是整个启动链路中**最重要的性能优化**。两个子进程启动后，CPU 转头去加载 ~200 个模块。当 `init()` 需要用到 MDM 设置和钥匙串凭证时，子进程很可能已经完成了——I/O 等待被计算时间完全覆盖。

### 5.3 模块导入广度

```typescript
// main.tsx 第 21-209 行（简化示意）
// 状态层       → bootstrap/state.js
// 配置层       → utils/config.js, utils/settings/*
// 遥测         → @opentelemetry/api, @growthbook/growthbook
// 工具系统     → Tool.js, tools.js
// 命令系统     → commands.js
// MCP          → services/mcp/*
// UI           → ink, react
// 权限         → utils/permissions/*
// 模型         → utils/model/*
// ... 总计约 190 个静态 import + 若干动态 require()
```

这些导入覆盖了应用的**所有子系统**。第 209 行的 `profileCheckpoint('main_tsx_imports_loaded')` 标记导入完成。

**CJS require() 的用途**：第 70-81 行使用了 `require()` 而非 ESM `import`：

```typescript
// main.tsx 第 70-72 行
const getTeammateUtils = () => require('./utils/teammate.js')
```

注释明确写了 "Lazy require to avoid circular dependency"。当模块图已无法用静态 ESM 分析表达时——例如 `teammate.ts → AppState.tsx → main.tsx → teammate.ts` 形成循环——CJS `require()` 的动态特性被用作逃生阀。这在架构上是**不得已的设计妥协**，而非风格偏好。

## 6. 阶段 4：核心初始化 `entrypoints/init.ts` 的 `init()`

### 6.1 职责

`init()` 是整个应用的**一次性初始化枢纽**。它在 Commander 的 `preAction` 钩子中被调用，通过 `memoize` 包装确保在整个进程生命周期中只执行一次。

```typescript
// init.ts 第 57 行
export const init = memoize(async (): Promise<void> => {
  // 20 步初始化...
})
```

### 6.2 初始化步骤表

> **实际运行步骤数少于 20 步**：以下带 `[fc]` 标记的步骤受 `feature()` 门控（build-time dead code elimination），外部构建中整块被移除。CCR 上游代理（步 15）还需要运行时环境变量 `CLAUDE_CODE_REMOTE=true` 才会触发。

| 步序 | 操作 | 检查点 | 说明 |
|------|------|--------|------|
| 1 | `enableConfigs()` | `init_configs_enabled` | 加载并合并多层配置 |
| 2 | `applySafeConfigEnvironmentVariables()` | — | 仅应用**安全**环境变量（信任前） |
| 3 | `applyExtraCACertsFromConfig()` | — | 在 TLS 握手前注入额外 CA 证书 |
| 4 | `setupGracefulShutdown()` | `init_after_graceful_shutdown` | 注册 SIGINT/SIGTERM 处理 |
| 5 | 动态导入并初始化 firstPartyEventLogger + growthbook | `init_after_1p_event_logging` | 遥测基础设施 |
| 6 | `populateOAuthAccountInfoIfNeeded()` | `init_after_oauth_populate` | OAuth 账户信息（非阻塞） |
| 7 | `initJetBrainsDetection()` | `init_after_jetbrains_detection` | 检测 JetBrains IDE（非阻塞） |
| 8 | `detectCurrentRepository()` | — | 检测当前 Git 仓库（非阻塞） |
| 9 | `initializeRemoteManagedSettingsLoadingPromise()` | `init_after_remote_settings_check` | 企业远程设置（条件触发） |
| 10 | `initializePolicyLimitsLoadingPromise()` | — | 策略限制（条件触发） |
| 11 | `recordFirstStartTime()` | — | 记录首次启动时间戳 |
| 12 | `configureGlobalMTLS()` | — | 全局 mTLS 证书配置 |
| 13 | `configureGlobalAgents()` | `init_network_configured` | 全局 Agent 配置 |
| 14 | `preconnectAnthropicApi()` | — | 预连接 TCP + TLS 握手 |
| 15 | 上游代理初始化 `[fc]` | — | CCR 环境专用，`feature('CCR')` 门控 |
| 16 | `setShellIfWindows()` | — | Windows 下设置 Shell |
| 17 | 注册 `shutdownLspServerManager` 清理 | — | 退出时关闭 LSP 服务器 |
| 18 | 注册 `cleanupSessionTeams` 清理 `[fc]` | — | 退出时清理团队会话 |

| 20 | `logForDiagnosticsNoPII('init_completed')` | `init_function_end` | 标记初始化完成 |

### 6.3 安全分阶段策略

注意第 2 步和第 3 步的顺序：**证书在环境变量之后**。

```
applySafeConfigEnvironmentVariables()  ← 信任前：只应用安全的子集
        │
        ▼ (信任对话框发生在此之后)
        │
applyConfigEnvironmentVariables()      ← 信任后：应用完整的 PATH、LD_PRELOAD 等
```

这是安全设计的关键——在用户确认信任之前，外部环境变量不会影响运行时行为。`applySafeConfigEnvironmentVariables()` 只设置 `NODE_EXTRA_CA_CERTS` 等不会引发安全问题的变量。

### 6.4 错误处理

```typescript
// init.ts 第 215-237 行
try {
  // 20 步初始化...
} catch (error) {
  if (error instanceof ConfigParseError) {
    // 显示配置错误对话框（Ink 组件）；非交互会话则直接输出到 stderr
    render(<InvalidConfigDialog error={error} />)
  } else {
    throw error  // 致命错误，终止启动
  }
}
```

`ConfigParseError` 是可恢复的——用户可以在对话框中修复配置。其他错误直接终止进程。

### 6.5 遥测延迟初始化

```typescript
// init.ts 第 247 行
export const initializeTelemetryAfterTrust = async () => {
  // 等待远程设置（如果符合条件）
  // 应用完整环境变量
  // 惰性加载 OpenTelemetry（~400KB + protobuf）
  // 初始化计数器 + 仪表
}
```

OpenTelemetry 的加载被推迟到**信任对话框之后**——换句话说，如果用户拒绝了信任，遥测代码根本不会被加载。即使在信任之后，它也是通过 `setImmediate` 延迟到首次渲染之后的。

## 7. 阶段 5：Commander 与 `run()` 的 `preAction`

### 7.1 职责

`main.tsx` 的 `run()` 函数（第 890-1100+ 行）创建 Commander 程序实例，并在 `preAction` 钩子（第 913-973 行）中完成"init 之后、action 之前"的初始化工作。

### 7.2 preAction 流程

`preAction`（第 913-973 行）按严格顺序执行以下步骤：

```typescript
// main.tsx preAction 钩子 第 913-973 行（简化）
program.hook('preAction', async () => {
  // 1. 等待阶段 3 启动的子进程完成（必须在 init() 之前，因为 init()
  //    内部会触发首次设置读取，MDM 未就绪将导致同步钥匙串读取 ~65ms）
  await Promise.all([
    ensureMdmSettingsLoaded(),
    ensureKeychainPrefetchCompleted(),
  ])

  // 2. 主初始化（阶段 4）
  await init()

  // 3. 设置进程标题（受 CLAUDE_CODE_DISABLE_TERMINAL_TITLE 门控）
  process.title = 'claude'

  // 4. 连接分析记录器（动态 import，对子命令也生效）
  const { initSinks } = await import('./utils/sinks.js')
  initSinks()

  // 5. 处理 --plugin-dir（对子命令也生效，因为子命令有独立 action）
  const pluginDir = thisCommand.getOptionValue('pluginDir')

  // 6. 运行设置迁移（幂等：检查旧键→写入新键→删除旧键，确保跨版本兼容）
  runMigrations()

  // 7. 加载远程托管设置 + 策略限制（非阻塞）
  void loadRemoteManagedSettings()
  void loadPolicyLimits()

  // 8. 后台上传用户设置（feature 门控）
  void uploadUserSettingsInBackground()
})
```

### 8.1 职责

`setup()` 负责一切**文件系统和环境层面**的准备工作。在 Commander `action` 处理程序中被调用，位于模型解析之前。

### 8.2 setup 流程

```typescript
// setup.ts 第 56 行（简化）
export const setup = async () => {
  // 1. Node.js 版本检查
  checkNodeVersion('>=18')

  // 2. 自定义会话 ID
  if (sessionId) switchSession(sessionId)

  // 3. UDS 消息传递（条件触发）
  if (UDS_INBOX) startUdsMessaging()

  // 4. 终端备份与恢复
  await restoreTerminalBackup()  // iTerm2 / Terminal.app

  // 5. 设置当前工作目录
  setCwd(cwd)

  // 6. 捕获钩子配置快照
  captureHookConfigSnapshot()

  // 7. 初始化文件变更监视器
  initFileChangedWatcher()

  // 8. 工作树处理（--worktree 标志）
  if (worktree) {
    await createWorktree()
    process.chdir(worktreePath)
    // 更新钩子快照（目录变了）
  }

  // 9. 后台任务
  initSessionMemory()
  initContextCollapse()
  lockCurrentVersion()

  // 10. 预取命令 + 插件钩子
  getCommands(getProjectRoot())
  loadPluginHooks()
  setupPluginHookHotReload()

  // 11. 发布说明检查
  checkReleaseNotes()

  // 12. 权限安全检查
  validateBypassPermissionsSafety()

  // 13. 上次退出事件记录
  logLastExitMetrics()
}
```

### 8.3 工作树处理（`--worktree` 标志）

这是 `setup()` 中最复杂的逻辑。当用户使用 `--worktree` 标志时：

```
创建 git worktree
  └─ 可选创建 tmux 会话
      └─ process.chdir() 到 worktree 路径
          └─ 更新 originalCwd
          └─ 更新 projectRoot
          └─ 更新 hookConfigSnapshot（新目录可能有自己的 .claude/ 配置）
```

这一切必须在所有文件相关逻辑（命令加载、Agent 定义、钩子注册）之前完成，因为这些逻辑依赖 `projectRoot` 和 `cwd` 的正确性。

### 8.4 预取策略

`setup()` 中的预取策略体现了"能用并行就不用串行"的原则：

```
setup()
  └─ getCommands()  ←──── 与 setup() 后续步骤并行
  └─ loadPluginHooks() ←── 与 setup() 后续步骤并行
```

这两个调用启动后，setup() 继续执行后续步骤，在需要命令和钩子结果时再 `await`。在非工作树模式下，这个并行化节省约 28ms。

## 9. 阶段 7：模型与 Agent 解析

### 9.1 职责

`setup()` 返回后，Commander 的 `action` 处理程序（位于 `main.tsx` 的 `main()` 函数内，约第 700-890 行）进入**模型和 Agent 解析**阶段。这是用户配置与运行时能力的交汇点。核心函数调用链为：

```

```

### 9.2 模型解析链路

```
--model 标志              ← 用户指定（最高优先级）
   └─ 无 → 用户设置       ← 配置文件中的默认模型
        └─ 无 → 系统默认  ← 硬编码兜底
              │

              │
              ▼

              │
              ▼
         顾问模型          ← --agent 标志指定的顾问
              │
              ▼
         最终模型 + 参差数
```

### 9.3 并行加载

```typescript
// main.tsx action 处理程序
const [commands, agentDefs] = await Promise.all([
```

命令和 Agent 定义的加载在 `setup()` 中就已启动，此时只需 `await` 结果。这是一种**流水线模式**——下一阶段在上一阶段结束前就启动了。

`--agents` 标志允许用户在启动时通过 JSON 定义临时 Agent。这在 CI/CD 环境（如 GitHub Actions）中非常有用——不需要修改任何配置文件。

## 11. 全局状态容器 `bootstrap/state.ts`

### 11.1 职责

`bootstrap/state.ts` 是整个应用的单例状态存储。它的文件开头有一行醒目的注释：

> **`DO NOT ADD MORE STATE HERE - BE JUDICIOUS WITH GLOBAL STATE`**

### 11.2 设计约束

`STATE` 是整个模块图的**叶子节点**——它被所有模块依赖，但它不依赖任何模块：

```
utils/  commands/  tools/  components/  services/
   │       │        │         │           │
   └───────┴────────┴─────────┴───────────┘
                     │
                     ▼
              bootstrap/state.ts  ← 被所有人导入
                     │
                     ▼
              (无依赖——纯 Node.js API)
```

这种设计防止了循环依赖。状态模块只使用 `node:path`、`node:crypto` 等 Node.js 内置模块，不导入任何项目内的模块。

### 11.3 状态字段分类

`getInitialState()` 初始化 ~70 个字段：

| 类别 | 字段示例 | 说明 |
|------|---------|------|
| 会话 | `sessionId`, `startTime`, `lastInteractionTime` | 会话标识与计时 |
| 路径 | `originalCwd`, `projectRoot` | 工作目录与项目根目录 |
| 客户端 | `clientType` | `'cli'` / `'github-action'` / `'sdk-*'` 等 |
| 交互性 | `isInteractive` | 是否为交互式会话 |
| 成本 | `totalCostUSD`, `modelUsage` | 计费统计 |
| 遥测 | `meter`, `sessionCounter` | OpenTelemetry 指针 |
| 标志 | 各种 Beta 标志和功能开关 | 运行时特性控制 |

### 11.4 访问器模式

`state.ts` 为每个字段提供配对的 getter/setter，**从不直接暴露 `STATE` 对象**。这确保了状态的任何修改都可追踪——调用者必须经过访问器函数，而非直接操作内部对象。模式高度规范：

- 简单字段：`getSessionId()` / `regenerateSessionId()` / `switchSession()`
- 计数器：`addToTotalCostState()` / `getTotalCostUSD()` / `getTotalInputTokens()`（通过 `sumBy` 聚合 `modelUsage`）
- 遥测：`setMeter()` 一次初始化 10 个 AttributedCounter
- 惰性刷新：`updateLastInteractionTime()` 标记脏位，`flushInteractionTime()` 在 Ink 渲染帧中统一更新，避免每次按键都调用 `Date.now()`

## 12. 关键设计决策（回顾）

> 以下 8 条设计原则在上文各阶段中已经展现，这里做集中回顾与总结。每条末尾以 `→ §X.Y` 标注对应的详细分析位置。

### 12.1 I/O 与计算重叠

```
时间线:
  ├─ startMdmRawRead() ──────────────────┤ (子进程，~100ms)
  ├─ startKeychainPrefetch() ────────────┤ (子进程，~80ms)
  ├─ ~200 个模块导入 ────────────────────┤ (CPU，~135ms)
                                         │
                                         ├─ await ensureMdmSettingsLoaded()
                                         ├─ await ensureKeychainPrefetchCompleted()
```

启动最慢的部分不是代码执行，而是 I/O 等待（MDM 策略、钥匙串凭证）。解决方案：在模块导入**之前**启动子进程，让 I/O 等待与计算重叠。这是整个启动链路中最大的性能优化——节省约 135ms。

### 12.2 快速路径模式

每个特殊操作都应该用最少的代码完成：

```
--version → console.log + process.exit (零导入)
--dump-system-prompt → ~3 个导入
--daemon-worker → ~3 个导入
正常会话 → ~200 个导入
```

这确保了用户执行简单操作时不需要等待巨型模块加载。`--version` 的响应时间接近终端原生命令。

### 12.3 懒加载策略

全链路懒加载，never `import`，always `await import()`：

```typescript
// cli.tsx: 只在需要时才加载 main.tsx
await import('../main.js')

// main.tsx: 只在需要时才加载 setup.ts
const { setup } = await import('./setup.js')

// main.tsx: 只在准备启动 REPL 时才加载交互式辅助
const { showSetupScreens, renderAndRun } = await import('./interactiveHelpers.js')
```

每一步都延迟到必需的时刻。这不仅加快了启动，也降低了每个快速路径的内存占用。

### 12.4 `memoize` 单例 `init()`

```typescript
export const init = memoize(async () => {
  // 只执行一次
})
```

`memoize` 保证 `init()` 在多次调用时（如测试、重连）不会重复执行。它在 `preAction` 钩子中被调用，所以每个 Commander 命令执行都会触发它——但实际工作只做一次。

### 12.5 安全分阶段环境变量

```
信任前: applySafeConfigEnvironmentVariables()
  └─ 仅设置: NODE_EXTRA_CA_CERTS 等"无害"变量

  ... 信任对话框 ...

信任后: applyConfigEnvironmentVariables()
  └─ 完整设置: PATH, LD_PRELOAD, 自定义环境变量
```

在用户确认信任之前，配置文件中的环境变量**不会**被注入到进程。这阻止了恶意配置通过环境变量劫持执行。

### 12.6 延迟遥测

OpenTelemetry 的 SDK（含 protobuf 序列化）约 400KB。在信任确认前加载它不仅浪费带宽，还可能导致用户在拒绝信任后仍然被追踪。因此：

1. 信任前：不加载 OTEL
2. 信任后：在 `setImmediate` 中加载（延迟到首次渲染之后）
3. 数据上报在首屏渲染完成后才开始

### 12.7 `bootstrap/state` 作为 DAG 叶子

通过将所有共享状态集中到一个不导入任何项目模块的单独文件中，Claude Code 避免了一类常见的架构问题——循环依赖。这是一个**中心辐射型（hub-and-spoke）**状态管理模式：

- 所有模块导入 `state.ts`（辐条）
- `state.ts` 不导入任何模块（这是一个单向依赖图）

### 12.8 启动性能分析

```typescript
profileCheckpoint('main_tsx_entry')
profileCheckpoint('main_tsx_imports_loaded')
profileCheckpoint('init_configs_enabled')
profileCheckpoint('init_after_graceful_shutdown')
profileCheckpoint('init_after_1p_event_logging')
profileCheckpoint('init_after_oauth_populate')
profileCheckpoint('init_after_jetbrains_detection')
profileCheckpoint('init_after_remote_settings_check')
profileCheckpoint('init_network_configured')
profileCheckpoint('init_function_end')
```

每个 `profileCheckpoint` 记录一个时间戳。在启动结束时，`profileReport()` 计算各阶段耗时并上报到 Statsig（内部用户 100% 采样，外部用户 0.5% 采样）。

**手动开启详细分析**：

```bash
CLAUDE_CODE_PROFILE_STARTUP=1 claude
```

这会生成一份完整的文件写入 + 时间线报告，用于诊断启动性能问题。

## 14. 总结

启动链路的本质是一个**分阶段、可中断、可并行的初始化管道**。每个阶段只做属于它那一层的事情：

- **dev-entry.ts** 确保项目完整，不具备完整性就不往下走
- **cli.tsx** 快速路由，不需要交互的就立即返回
- **main.tsx 顶级** 启动 I/O，让等待与计算重叠
- **init.ts** 一次性初始化，信任前不做危险操作
- **setup.ts** 搭环境、创建工作树、启动后台任务
- **action 中段** 解析用户配置、确定模型和 Agent
- **action 尾段** 渲染 UI、启动预取、REPL 就绪

理解了这条链路，就理解了 Claude Code **从命令行到 REPL** 的全部秘密。
