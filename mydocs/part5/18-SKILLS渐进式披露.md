# 18. SKILLS 渐进式披露——技能的注册、执行与注入

## 1. 什么是渐进式披露

渐进式披露是 Harness 管理"模型每个回合能知道什么"的设计模式——不把全部知识一次性塞给模型，而是在每一轮对话中分层、按需、增量地呈现，在信息量、缓存命中率和上下文预算之间取得折衷。

**两条不可改变的约束**

**上下文窗口是硬上限。** 200K token 的空间要同时容纳 system prompt、用户消息、工具结果。如果技能清单占用了过多空间，留给用户问题和工具输出的预算就会急剧压缩——用户多问两句就可能触发压缩。

**Prompt Cache 有经济账。** Anthropic API 的缓存按前缀匹配：system prompt 越长、变化越频繁，命中率越低。一次 cache miss 意味着额外的 token 写入（按 cache_creation 计费）和更长的响应延迟，缓存命中率下降会显著推高会话的整体成本。

**渐进式方案演进**

最早尝试**全量注入**，所有技能一次性塞进 system prompt。模型什么都知道，但上下文预算被技能清单挤占。后来转向**RAG 按需检索**，预算被解放，但模型什么都不知道，只能在黑暗中猜测。

渐进式披露走中间路线：**先给目录（保有可见性），再按需给详情（控制预算）。**


```mermaid
flowchart LR
    A["全量注入"] -->|"❌ 上下文爆炸"| B["RAG 按需检索"]
    B -->|"❌ 模型不知有何可用"| C["渐进式披露 ✓"]
    C --> D["先注册再注入<br/>先目录再详情"]
```


两个核心问题：

- **为什么要"渐进"？**——上下文预算和缓存的刚性格局
- **渐进式披露怎么分步走？**——先拆 system prompt、再把变化内容移出去、最后用增量收尾，以及技能系统如何接入

## 2. 核心逻辑

### 2.1 搭建三层注入管线

有了动机，下一个问题是"怎么实现"。显然不能全写进一个 system prompt 字符串，需要分层。

这套管线不是从一开始就是三层结构。它经历了一个演化过程：先是发现整块 system prompt 不行，拆成两块；再发现两块还不够，把变化的内容移出去；最后发现变化的内容也需要管理粒度。**每一层都在修补上一层的不足。**

#### 2.2.1 第一层：System Prompt 分层组装

`getSystemPrompt()` 将所有输出分为**静态前缀**和**动态后缀**，中间用 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 标记分界线。静态段不变，跨会话共用缓存；动态段依赖对话上下文，轮次内缓存——技能系统的注入就发生在动态段。

动态段通过 **section 注册表**统一管理：

- `systemPromptSection(name, compute)` 注册可缓存段，技能用法提示等注册为此类，计算后缓存直到 `/clear` 才失效
- `DANGEROUS_uncachedSystemPromptSection(name, compute, reason)` 注册不可缓存段，每轮重新计算

每轮对话先检查缓存命中，没有再调用 compute。技能目录的格式化结果在技能注册表不变时直接取缓存，无需重算。

第一层把变化范围限制在动态段内部。但技能列表会随会话推进增长（如新插件注册技能），每次变化都会打穿整个动态段的缓存——即使变化只发生在技能段落，其他动态段也被迫重算。于是需要第二层：把变化频繁的技能列表从 system prompt 中移出去，交给独立的附件通道。

#### 2.2.2 第二层：附件通道

既然变化频繁的内容放在 system prompt 里没法避免影响缓存，那就把它们从 system prompt 中移出去。`<system-reminder>` 附件机制为此而生。

附件通过工具结果或用户消息注入到对话中，它不是 system prompt 的一部分，不参与 system prompt 的缓存计算。这意味着附件内容的增减、变化，完全不影响 system prompt 的缓存前缀。这就是第二层的核心思路：**按稳定性分离，变化的内容走独立的通道。**

对于技能系统，三个核心附件类型：

| 附件类型 | 触发时机 | 作用 |
|----------|---------|------|
| `skill_listing` | 会话启动 + 新技能注册时 | 把可用技能列表（按预算裁剪）发给模型 |
| `dynamic_skill` | 文件操作发现新 `.claude/skills/` | 通知模型当前目录有新技能可用 |
| `invoked_skills` | 压缩发生后 | 提醒模型已经调用过哪些技能 |

**预算裁剪**是附件通道的关键配套。如果技能列表本身就有几百条，即使通过附件注入，模型也看不完。`formatCommandsWithinBudget()` 按 context window 的 1% 计算字符预算——例如 200K 窗口对应约 8,000 字符。在这个预算内，bundled（内置）技能始终完整展示，因为它们是所有会话的基础能力。非 bundled 技能按优先级逐级降级：先裁到描述不超过 250 字符，再裁到仅保留技能名称。

```
总字符预算 = contextWindow × 4 (chars/token) × 1%
         ≈ 8,000 (200K 窗口)

分配逻辑：
  1. Bundled 技能优先：完整展示（name + description + whenToUse）
  2. 非 bundled 技能：先尝试完整展示
  3. 超预算：裁剪 description 至均分长度
  4. 仍超预算：非 bundled 技能降级为名称展示
```

附件通道也不是每轮全量重发。系统维护一个"已发送技能集合"，新生成的 `skill_listing` 会先和已发送列表做差集——只发送新增的。如果没有任何变化，`skill_listing` 附件就不会被生成。这个去重减少了每次注入的变化量。

> **设计意图：** system prompt 只放稳定的，变化的交给附件。附件也不是全量重发——"只发新增"比"每次都发完整列表"更进一步缩小了变化范围。

#### 2.2.3 第三层：Delta 增量

对于变化特别频繁的信息，连附件通道的"增量发送"都不够——因为增量还是有变化量，每次变化都可能让 Anthropic 的**后缀缓存**失效。更精细的粒度是只发送变化的部分本身，而不是"完整的新内容"。

Delta 通道实现了这一点。它不是重新发送某个附件的完整内容，而是只发送 diff：

| Delta 类型 | 内容 |
|-----------|------|
| skill_listing 去重 | 第一轮发全部，后续只发新增或变化的技能 |
| `mcp_instructions_delta` | MCP 服务器连接/断开时只发指令的差异 |
| `agent_listing_delta` | Agent 类型变化时只发差异 |
| `deferred_tools_delta` | 工具懒加载时的增量通知 |

以 `skill_listing` 的去重为例：初始会话时，第一轮 `skill_listing` 包含全部可用技能。后续如果用户安装了一个新插件，插件注册了新技能，第二轮 `skill_listing` 不会重新发送完整的技能列表——它只发送新加入的那 2-3 个技能的名称和描述。模型在收到这个增量后，将其追加到自己对技能目录的认知中。如果有技能被禁用或删除，则发送"移除列表"。

**三层的演进关系：**

```
变化量 ↓                   缓存命中率 ↑
─────────────────────────────────────────→
System Prompt 静态+动态段
         → 附件通道承载变化内容
                 → Delta 最小化单次变化
```

每前进一步，单次注入的变化量就缩小一个量级，缓存命中率相应提高。代价是系统的复杂性同步增加——需要维护去重状态、delta 生成逻辑、以及各通道间的调度协调。并不是所有内容都需要走到第三层。只有那些变化特别频繁的内容才值得付出额外的管理成本进入 Delta 通道。大部分内容停在第一层或第二层就足够了。

> **设计意图：** 三层管线不是初始设计的，而是在"缓存又失效了"这个反馈循环中逐步演化出来的。每一层都在回答同一个问题：怎么让这次的变化比上次更小？当变化小到可以忽略不计时，系统的缓存命中率就达到了最优。

### 2.3 技能系统的渐进式注入闭环

三层管线解决了"怎么展示可用知识"的问题。但管线本身不产生知识——它的水源是技能系统。没有技能系统，三层管线就没有内容可送。反过来，没有三层管线，技能系统只能把全部技能描述塞进 system prompt，效果大打折扣。**两者互为前提。**

技能从注册到注入到留存的完整生命周期，恰好验证了三层管线设计是否成立。

#### 2.3.1 注册：有什么可披露

模型能用的技能从六个来源汇集。不是所有来源在一开始就存在，它们是随着使用场景的扩展逐步加入的：

- **Bundled**：编译时注册，随 CLI 发布。始终可用，也是唯一不需要文件系统扫描的技能来源。
- **Disk**：从 `.claude/skills/` 目录扫描 SKILL.md 文件加载。用户、项目、企业各有独立的管理目录，互不干扰。
- **Plugin**：通过插件分发机制安装的技能。插件可以打包多个技能，统一管理版本和更新。
- **MCP**：远程 MCP 服务器提供的 PromptCommand。技能内容不在本地，而在远程服务器上。
- **Dynamic**：在文件操作时，从当前文件路径向上遍历目录，发现嵌套的 `.claude/skills/`。
- **Conditional**：声明了 `paths` 条件的技能。它们注册了但不在激活状态，直到用户编辑匹配路径的文件时才被激活。

多源并行必然带来重复。去重的基准不是文件名，而是 `realpath` 解析后的真实路径。选择 realpath 而非 `fs.stat().ino`，是因为某些虚拟文件系统（如 Docker 容器、NFS）返回的 inode 为 0，而 realpath 在所有平台下行为一致。同一条路径的多个来源按优先级取第一个：managed > user > project > plugin。

条件技能是最彻底的"渐进"实践。它注册了但不激活，不占用注入管线的任何预算。用户 clone 了一个包含 Python 技能的项目，但直到他编辑第一个 `.py` 文件，系统才知道需要激活这个技能。在那之前，这个技能不存在于模型的视野中。**注册和披露是完全分离的两个阶段。**

#### 2.3.2 展示：Skill Tool 的双重角色

注册完了，模型怎么知道有哪些技能可用？系统通过 Skill Tool 向模型展示技能目录。

但 Skill Tool 不是一个单纯的"展示面板"。它在系统中扮演了两个截然不同的角色，对应着"模型能看到什么"和"模型能用什么"两个问题。

**第一角色——目录展示。** Skill Tool 的 prompt 通过 `formatCommandsWithinBudget()` 动态生成可用技能列表。这个列表不是写死在 system prompt 里的——它通过附件通道的 `skill_listing` 注入。每次技能注册表有变化时，系统对比当前技能集合和上次发送的集合，生成增量 diff，再通过附件通道推送给模型。

模型看到的 `/skill-name` 列表，本质上是一个**经过三道过滤的结果**：
1. **注册过滤**：只有注册过的技能才会进入备选池
2. **优先级过滤**：非 bundled 技能在预算不足时降级或截断
3. **时机过滤**：条件技能在没有命中路径前不会出现

**第二角色——调用闸门。** 模型选择了某个技能后，Skill Tool 不直接执行，而是先验证再调度。

验证阶段关注三个问题：技能是否存在、当前上下文是否允许调用、是否有权限约束。验证的结果不是二元的"通过/拒绝"——它把决定权交给权限系统，支持三态：允许、拒绝、交由用户决定。

调度阶段决定执行模式：inline 还是 fork。这个选择取决于技能声明的 `context` 字段和内容量与预期 token 消耗。

> **设计意图：** 展示窗口 + 执行闸门，两个角色让 Skill Tool 成为了渐进式披露在技能侧的**唯一入口**。它既要控制"模型的认知范围"，又要控制"模型的行动范围"。

#### 2.3.3 执行：inline 注入与 fork 分流

通过了闸门，技能内容以什么方式进入对话？这个问题比看起来复杂——技能内容可能只有几行提示，也可能包含完整的验证脚本和测试用例。不同的内容量需要不同的注入策略。

**Inline（默认模式）。** 技能的完整 content 直接注入当前对话。这个注入过程不是简单的字符串拼接——它包含三个预处理步骤：

- **参数插值**：将模型传入的 arguments 替换 content 中的 `${ARGUMENTS}` 占位符。这是技能实现参数化的基础——同一个技能根据传入参数的不同生成不同的注入内容。
- **目录变量**：`${CLAUDE_SKILL_DIR}` 替换为技能文件所在目录的绝对路径。技能中可以引用相对路径下的辅助文件，不必硬编码路径。
- **Shell 展开**：如果 content 中包含 `!` 前缀的命令行或 ` ```! ` 代码块，系统在执行注入前先执行这些 shell 命令，将输出嵌入 content。这使得技能可以包含动态生成的内容。

Inline 的代价是技能内容占用主对话的上下文预算。模型调用一次技能，技能的全部 content 就进入了对话历史，再也不退出。如果这个技能内容有 2K token，而且模型在会话中反复调用 10 次，那就是 20K token 的固定开销。对于大多数轻量技能（如 `commit`、`test`）来说这个开销可以接受，但对于重型任务来说需要另一种模式。

**Fork（隔离模式）。** 声明了 `context: fork` 的技能在子 Agent 中执行。子 Agent 拥有独立的 token 预算和 tool 上下文，它的整个执行过程与主对话隔离。主对话只看到工具调用的结果（一段文本输出），看不到子 Agent 内部的思考过程和中间调用。Fork 模式下，重型技能的 token 消耗不会冲击主对话的预算，中间过程不会污染主对话的上下文。

Inline 和 Fork 的选择本身也是一种"渐进"：轻量知识 inline 注入，不影响缓存结构；重型任务 fork 隔离，不污染主对话预算。

> **设计意图：** Inline 是"分期付款"——只用的时候才扣预算；Fork 是"转移支付"——不让大额消费影响主账户流水。两种模式让技能系统既能轻又能重，而选择权交给技能的声明者，而非调用者。

#### 2.3.4 注入后的生命周期管理

技能从注册到注入到留存，不是一条直线，而是一个反馈闭环。三层注入管线中的所有通道都需要在正确的时间点被激活：

| 时机 | 动作 | 对应机制 |
|------|------|:---:|
| 会话启动 | `skill_listing` 附件发送可用技能目录 | 附件通道 |
| 日常对话 | System Prompt 动态段中的 `/skill-name` 用法提示 | System Prompt 动态段 |
| 文件操作 | 遍历目录 → 发现 `.claude/skills/` → `dynamic_skill` 通知 | 附件通道 |
| 条件匹配 | paths 命中 → 条件技能激活 → 加入技能注册表 | 附件通道 |
| 模型调用 | Skill Tool 验证 → inline/fork 分流 → 加载 content → 注入对话 | 执行层 |
| 压缩之后 | 通过 `invoked_skills` 附件重注入已调用技能，防止上下文丢失 | 附件通道 |

最后一个时机——压缩后——值得特别解释。当对话接近 token 上限时，系统会对早期消息进行压缩（这是 Part 6 的内容）。压缩后，那些早期被调用的技能内容可能已经不在上下文中了。如果不做任何处理，模型会忘记自己曾经用过哪些技能。`invoked_skills` 附件就是为此服务的：它记录了这个会话中调用过的所有技能，在压缩后重新注入，提醒模型"这些是你用过的"。

这不是技能系统独有的。其他知识类型复用同一套机制：

- **Memory**：通过 `systemPromptSection('memory')` 走 System Prompt 动态段注入。Memory 的内容相对稳定，放在 system prompt 中不会频繁变动，因此不需要走附件通道。
- **MCP 指令**：走 `mcp_instructions_delta` 或附件通道。MCP 服务器的连接和断开是运行时事件，变化频繁，适合走更细粒度的 Delta 通道。
- **CLAUDE.md 规则**：通过附件通道的 `activated_rules` 注入。规则依赖于当前操作的文件路径，类似条件技能。

这些不同类型的知识共用同一套管道，但各自根据变化频率选择了不同的管段——Memory 停在第一层，MCP 指令走到第三层，技能系统用了所有三层。**管道的分段设计允许知识按照自身的变化特性选择合适的暴露粒度。**

>

> **设计意图：** 技能系统最完整的生命周期验证了这套管线的可行性。但更重要的是，它为未来的知识类型铺好了路——任何"需要让模型知道但不需要每轮都知道"的信息，都可以接入这套渐进披露体系，不需要重新设计注入机制。
