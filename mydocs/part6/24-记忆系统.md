# 24. 记忆系统-跨会话记忆

[第 22 章](../part6/22-三层压缩策略-上-Micro-Compact与工具结果清理)和[第 23 章](../part6/23-三层压缩策略-下-Compact对话压缩与摘要)解决的是「上下文窗口满了怎么办」——用免费的工具结果清理和付费的 LLM 摘要，把历史对话压缩成可管理的尺寸。但压缩只是腾出了空间，并没有回答一个更根本的问题：**当对话结束、会话关闭，下一次打开时，Agent 还记得什么？**

答案就在本章——记忆系统，Claude Code 的跨会话持久记忆方案。它在源码中的内部代号是 Memory Directory（memdir）。

## 1. 背景介绍

### 1.1 跨会话的持久记忆

前面的三层压缩策略，无论 Micro-Compact 还是 Auto-Compact，处理的都是**当前会话内**的上下文问题。窗口快满了？清掉过时工具结果。还不够？用 LLM 把对话总结成摘要。但所有这些操作都有一个隐含的假设：**上下文的生命周期等于会话的生命周期。** 会话结束，上下文消失。

现实并非如此。用户和 Claude Code 的协作往往跨越多次会话——今天修了一个 bug，明天继续同一个功能；这周定下的技术决策，下周还需要遵守。如果每次打开 Claude Code 都像「初次见面」，用户就得反复解释自己的背景、偏好、项目状态——这不仅是体验问题，更是效率问题。

记忆系统解决的就是这个「跨会话」问题。它的定位可以从三个维度理解：

```
会话内                          跨会话
┌──────────────────┐    ┌──────────────────┐
│ Compact (第23章)  │    │ Memdir (本章)     │
│ 临时压缩，会话结束 │    │ 持久存储，下次会话  │
│ 即丢弃             │    │ 自动加载          │
└──────────────────┘    └──────────────────┘

LLM 生成                      文件系统存储
┌──────────────────┐    ┌──────────────────┐
│ Compact 摘要      │    │ Memdir           │
│ 模型生成的摘要文本 │    │ .md 文件，人类可读 │
│ 不可手动编辑      │    │ 可手动编辑、git 追踪│
└──────────────────┘    └──────────────────┘
```

关键洞察：记忆系统不是 Compact 的替代品，而是**不同时间尺度的互补系统**。Compact 让 Agent 在会话内不迷路，记忆系统让 Agent 在会话之间不归零。

### 1.2 什么值得记住，什么不该记？

有了「跨会话持久化」的目标，第一个棘手的决策不是技术方案，而是**边界——什么信息值得跨会话保留？**

先看极端。一端是「什么都记」——每轮对话、每次工具调用结果、每个文件改动都存下来。结果是记忆库迅速膨胀成垃圾场，检索成本飙升，信噪比趋近于零。另一端是「什么都不记」——每次会话从零开始，用户反复解释同样的偏好和背景。

折衷方案藏在一个简单的判断中：**代码库已经能回答的信息，不需要记。** 这是记忆系统最核心的设计原则。源码中有明确的表述：

> Memories are constrained to four types capturing context NOT derivable from the current project state. Code patterns, architecture, git history, and file structure are derivable (via grep/git/CLAUDE.md) and should NOT be saved as memories.

Agent 手上有 Grep、Glob、Git 等工具，代码结构、架构模式、提交历史都可以在需要时实时查询——它们是「可推导」的信息，占用记忆空间是纯粹的浪费。真正值得跨会话保留的，是那些**工具查不到的信息**。系统将这些信息收敛为四种类型：

| 类型 | 存什么 | 为什么工具查不到 |
|------|--------|-----------------|
| `user` | 用户角色、技能、偏好 | "我是数据科学家"——代码里没有这句话 |
| `feedback` | 协作方式的纠正和确认 | "别 mock 数据库，上次 mock 通过但生产迁移失败"——git log 不会记录这个教训 |
| `project` | 进行中的工作、决策、截止日期 | "合并冻结 3 月 5 日开始"——不是代码状态，是人的计划 |
| `reference` | 外部系统的指针 | "pipeline 的 bug 在 Linear 项目 INGEST 里跟踪"——代码仓库之外的信息 |

反过来，也有明确的「不该记」清单。这条规则甚至在用户明确要求保存时也适用——如果用户让你保存 PR 列表，你应该追问「其中什么让你意外？」，因为 PR 列表本身可以从 git 推导：

> These exclusions apply even when the user explicitly asks you to save. If they ask you to save a PR list or activity summary, ask what was *surprising* or *non-obvious* about it — that is the part worth keeping.

这引出一个自然的疑问：CLAUDE.md 也是文件系统上的 markdown，也注入 system prompt，也影响 Agent 行为。它和记忆系统是什么关系？答案是：**角色不同，但边界互补。**

| | CLAUDE.md | 记忆系统 |
|---|---|---|
| **谁写** | 人（项目维护者） | Agent（自动提取，或对话中主动写入） |
| **内容性质** | "怎么做"——编码规范、项目约定 | "发生了什么"——用户偏好、项目背景、外部指针 |
| **生命周期** | 随代码版本控制 | 随会话自动演进 |
| **存放位置** | 项目根目录 / `~/.claude/` | `~/.claude/projects/<slug>/memory/` |

一句话区分：CLAUDE.md 是说明书，记忆系统是日记本。说明书告诉 Agent 怎么做事，日记本告诉 Agent 之前发生过什么。

划定了记忆的边界，接下来的工程问题就是：这些记忆以什么结构存到文件系统？怎么在合适的时机写进去？需要时怎么精确地取出来？

## 2. 核心逻辑

### 2.1 怎么组织记忆文件？

有了信息分类标准，下一个问题是物理存储。约束条件很清楚：

- 记忆数量会持续增长（几十到上百条）
- 上下文窗口有限（每次会话只能加载一小部分）
- 模型写入记忆是通过 tool call（Write/Edit 工具）
- 人类可能需要手动编辑或删除某条记忆

**方案演进**：

| 方案 | 问题 |
|------|------|
| 所有记忆放在一个大文件里 | 文件随记忆增长线性膨胀，每次加载全部内容 → token 浪费。模型修改一条记忆需要 Edit 整个文件。 |
| 纯文件系统，每个文件一条记忆，无索引 | 模型每次想「回忆」都得先 `ls` 再逐个 `read` frontmatter → 浪费 tool call 轮次。而且模型无法一次性建立「有哪些记忆」的全景认知。 |
| **MEMORY.md 索引 + 独立 topic 文件** | 目录文件始终全量加载（≤200 行、≤25KB），提供全景认知；topic 文件按需读取，避免浪费。 |

最终采用的二层结构：

```
~/.claude/projects/<sanitized-git-root>/memory/
├── MEMORY.md              ← 索引（一行一条，始终全量加载）
├── user_role.md           ← topic 文件（正文，按需加载）
├── feedback_testing.md
├── project_merge_freeze.md
└── ...
```

MEMORY.md 的格式极其简单——每行一个链接加一句描述：

```markdown
- [User role](user_role.md) — 用户是数据科学家，当前关注可观测性
- [Testing feedback](feedback_testing.md) — 集成测试必须连真实数据库
```

索引本身不包含记忆正文，正文全在 topic 文件中。这个设计有三个好处：

1. **索引轻量**：MEMORY.md 限制 200 行 / 25KB，任一触发即截断并附加警告——「索引太长了，请合并或归档老旧记忆」。加载成本有硬上界。
2. **写入友好**：模型保存记忆只需两步——先写 topic 文件，再在 MEMORY.md 加一行指针。两个独立的 tool call，互不干扰。
3. **人类可读**：每一条记忆是一个标准 markdown 文件，可以手动编辑、删除，甚至纳入 git 版本控制。

topic 文件的格式也做了标准化——frontmatter 声明元信息，正文承载内容：

```markdown
---
name: user-role
description: 用户是数据科学家，当前关注可观测性/日志
type: user
---

用户是数据科学家，当前主要工作是调查生产环境的日志覆盖情况。
在解释代码时优先关注可观测性相关的细节。
```

`type` 字段确保记忆类型不丢失——这是后续按需召回时做相关性判断的关键输入。

### 2.2 记忆怎么存进来？

结构定了，内容从哪来？这里有一个关键洞察：**模型在对话过程中已经「看到」了值得记忆的信息，让它顺便写下来，增量成本为零。**

但现实不是每次都完美——模型可能忙于执行任务而忘记记录。于是系统设计了**双轨写入**：主 Agent 主动写（零成本）+ 后台 Agent 补漏（有成本），两条路径互斥，避免重复花钱。

```
每轮 query loop 结束
  │
  ├─ 主 Agent 在这轮对话中写了记忆文件？
  │    │
  │    ├─ 是 → hasMemoryWritesSince() = true
  │    │       后台提取跳过此轮（免重复付费）
  │    │
  │    └─ 否 → 后台提取启动
  │            fork 一个 Agent，分析对话
  │            提取遗漏的记忆
  │            写入 memdir 目录
```

#### 路径 A：主 Agent 主动写入

System prompt 中注入了完整的记忆保存指令——四种类型的定义、什么时候该写、怎么写（两步法）、什么不该写。这些指令来自 [memoryTypes.ts](https://github.com/binarylei/claudecode/blob/main/src/memdir/memoryTypes.ts) 的模板和 [memdir.ts](https://github.com/binarylei/claudecode/blob/main/src/memdir/memdir.ts#L199-L266) 的组装逻辑。

因为指令已经在 system prompt 里了，模型在对话中判断「该记住什么」并直接调用 Write/Edit 工具——**不需要额外的 API 调用，增量成本为零。** 这是「能免费就不付费」的第一次体现。

#### 路径 B：兜底写入

路径 A 依赖模型的主动性——但模型可能因为任务繁忙而遗漏。路径 B 是安全网：每轮 query loop 结束后，系统 fork 一个 Agent 专门分析对话，提取遗漏的记忆。

**互斥机制**是这个设计的核心。在启动提取之前，系统会检查这一轮对话中主 Agent 是否已经写过记忆文件（[extractMemories.ts:121-148](https://github.com/binarylei/claudecode/blob/main/src/services/extractMemories/extractMemories.ts#L121-L148)）：

- 如果主 Agent 写了 → `hasMemoryWritesSince()` 返回 `true` → 跳过提取，光标前进
- 如果主 Agent 没写 → 启动提取，分析对话，找出值得保存的信息

为什么需要这个互斥？因为主 Agent 已经「免费」写过的东西，再花一次 API 调用去提取就是纯粹的浪费。互斥确保两条路径不会重复覆盖同一段对话。

**缓存共享让路径 B 尽可能便宜。** 后台提取使用 `runForkedAgent`——fork 出的 Agent 复用主 Agent 的 system prompt、工具列表和对话历史前缀，使其 prompt cache 命中率达到 98% 以上。这意味着路径 B 虽然是「付费」的，但实际开销被压到极低——大部分输入 token 走缓存，只支付增量部分的费用。这是「能免费就不付费，付费也要省着花」的第二次体现。

#### 提取策略

后台兜底有精细的发送控制：

- **节流**：不是每轮对话都触发。通过 `tengu_bramble_lintel` feature gate 控制间隔（默认每 1 轮）
- **合并**：如果上一轮提取还在进行中，新的提取请求不会立即执行，而是暂存为 `pendingContext`。当前提取完成后，自动执行一次「追赶提取」（trailing run），处理积压的对话
- **预算控制**：每次提取限制最多 5 轮 tool call，防止 Agent 陷入验证循环——「读到 → 写入 → 停止」，不允许去 grep 源码验证记忆的真实性

### 2.3 记忆怎么查出来？

记忆存进去是为了将来能用上。召回的设计同样遵循「能免费就不付费」的原则。

```
会话启动                      每轮 query
┌──────────┐                ┌──────────────┐
│ MEMORY.md │  全量注入       │ 用户 query    │
│ 索引全文   │  system prompt │ 当前对话上下文 │
│           │                │              │
│ 成本: 0   │                │ 成本: 1 次     │
│           │                │ 轻量 API 调用  │
└──────────┘                └──────┬───────┘
                                   │
                          Sonnet sideQuery
                          从所有 topic 文件中
                          挑选 ≤5 个最相关的
                                   │
                          ┌───────▼───────┐
                          │ 选中的 topic   │
                          │ 全文注入消息流  │
                          │ <system-reminder>│
                          └───────────────┘
```

#### 第一层：全量索引预加载（启动时，零成本）

System prompt 拼装时，[prompts.ts:495](https://github.com/binarylei/claudecode/blob/main/src/constants/prompts.ts#L495) 通过 `systemPromptSection('memory', ...)` 注入记忆系统指令和 MEMORY.md 索引全文。模型在会话启动时就「知道有哪些记忆存在」，但不需要读取每条记忆的正文。

这里有一个重要的工程细节：`systemPromptSection` 使用缓存策略——一次加载后整个会话期间复用。所以索引加载只在会话启动时发生一次，后续每轮对话零额外开销。

#### 第二层：按需相关性筛选（每轮 query 时，1 次轻量 API）

模型知道「有什么」，但不知道「每条的具体内容」。当用户的问题可能涉及某条记忆时，系统启动第二层召回。

[findRelevantMemories.ts](https://github.com/binarylei/claudecode/blob/main/src/memdir/findRelevantMemories.ts) 的工作流程：

1. 扫描记忆目录，读取所有 topic 文件的 frontmatter（文件名 + description + type + mtime），最多 200 个
2. 将用户 query、可用记忆清单、最近使用的工具列表发送给 Sonnet 小模型
3. Sonnet 返回 ≤5 个最相关的文件名
4. 系统读取这些文件全文，以 `<system-reminder>` attachment 形式注入消息流

关键设计决策——挑选标准是「确定相关才选」，而非「可能相关就选」：

> Only include memories that you are certain will be helpful based on their name and description. If you are unsure if a memory will be useful, then do not include it.

宁可漏报，不要误报。因为误报的代价不是一次 API 调用——而是大量无关记忆被注入上下文窗口，挤占真正的对话空间。

还有一个细节：如果最近使用的工具列表中包含某个工具，系统不会选取该工具的「使用说明」类记忆（因为 Agent 已经在用了），但仍会选取该工具的「警告、陷阱、已知问题」类记忆——因为正在使用时恰恰是这些问题最可能触发的时候。

#### 防过期：记忆是快照，不是真相

召回记忆只是第一步，**怎么用**才是关键。记忆说「某个文件存在」「某个函数可以调用」——但那是写记忆时的事实，不是现在的事实。

系统对此有三层设计：

1. **人性化时间**（[memoryAge.ts:15-20](https://github.com/binarylei/claudecode/blob/main/src/memdir/memoryAge.ts#L15-L20)）：展示「47 days ago」而非 ISO timestamp——模型对自然语言时间更敏感，比日期计算更能触发谨慎推理。

2. **过期警告**（[memoryAge.ts:33-42](https://github.com/binarylei/claudecode/blob/main/src/memdir/memoryAge.ts#L33-L42)）：超过 1 天的记忆自动附加 `<system-reminder>`："This memory is N days old. Memories are point-in-time observations, not live state — claims about code behavior or file:line citations may be outdated. Verify against current code before asserting as fact."

3. **验证指令**（[memoryTypes.ts:240-256](https://github.com/binarylei/claudecode/blob/main/src/memdir/memoryTypes.ts#L240-L256)）：system prompt 的 "Before recommending from memory" 节告诉模型——记忆声称某个文件存在 → 先 grep 验证。记忆声称某个函数可用 → 先读源码确认。"The memory says X exists" is not the same as "X exists now."

这组设计要防止的问题不是技术性的，而是心理性的：记忆文件带着 frontmatter、时间戳和详细的描述，看起来「很有权威感」。模型容易把它当作事实而非快照——验证指令就是专门打破这种权威幻觉。

### 2.4 会话摘要：记忆与压缩的桥梁

以上三层——组织、写入、召回——构成了记忆系统的闭环。但还有一个上游系统值得关注：Session Memory（会话摘要）。它是一份实时维护的 markdown 文件，Compact 触发时可直接复用为摘要，省下一次 LLM 调用。

```
会话摘要 (Session Memory)  →  Compact  →  后台提取  →  记忆目录
 实时记录会话关键信息         零成本摘要    补漏写入     持久记忆
```

**它是什么**

Session Memory 是一份**实时维护的 markdown 笔记文件**，按固定模板组织，记录会话中发生的重要事件——正在做什么、遇到了什么错误、哪些文件是关键、用户纠正了什么。

它按固定模板记录当前状态、错误与修复、关键文件、操作日志等内容（[prompts.ts:11-41](https://github.com/binarylei/claudecode/blob/main/src/services/SessionMemory/prompts.ts#L11-L41)），Compact 触发时可直接将其用作结构化摘要。

目录结构如下：

```
~/.claude/projects/<sanitized-git-root> ← 按 git 仓库隔离（跨 worktree 共享）
├─ memory/                       ← 记忆目录：跨会话持久记忆
│  ├─ MEMORY.md                  ← 索引（始终全量加载）
│  └─ user_role.md               ← topic 文件（按需召回）
│
├─ <session-id>.jsonl            ← 会话日志：完整对话记录
└─ <session-id>/                 ← 单次会话目录
   └─ session-memory/summary.md  ← 会话摘要：Compact 零成本替代
```

记忆目录用 git 根路径做隔离键——同一仓库的不同 worktree 共享记忆；会话日志和会话摘要用工作目录 + 会话 ID 做隔离键——每次会话独立存储。

**触发时机与复用**

它作为 post-sampling hook 注册（[sessionMemory.ts:357-375](https://github.com/binarylei/claudecode/blob/main/src/services/SessionMemory/sessionMemory.ts#L357-L375)），在每轮 API 响应后自动检查是否需要更新。触发条件（[sessionMemory.ts:134-181](https://github.com/binarylei/claudecode/blob/main/src/services/SessionMemory/sessionMemory.ts#L134-L181)）是双阈值 AND 逻辑：

- Token 增长 ≥ 5000（自上次更新以来）
- Tool call ≥ 3 次（自上次更新以来）
- 且上一条 assistant 消息没有 tool call（避免在工具调用中途打断）

首次触发还需要上下文总 token 数 ≥ 10000（初始化阈值）。

更新是异步的、forked agent 执行的——在 Agent 等待用户输入时后台完成，增量成本为零。当 Auto-Compact 触发时，系统优先尝试复用 Session Memory 作为摘要（[sessionMemoryCompact.ts:514-543](https://github.com/binarylei/claudecode/blob/main/src/services/compact/sessionMemoryCompact.ts#L514-L543)）：有内容则直接使用，省掉一次 Compact LLM 调用；仍为空模板则回退到传统 LLM 摘要。

## 3. 源码解读

### 3.1 源码地图：一张图走完"加载→写入→召回→复用"全流程

记忆系统涉及 13+ 个源文件，代码分布较广，但核心调用路径围绕两个顶层时间点展开——启动时加载索引，每轮 query 循环中按「API 调用前 → API 调用中 → API 调用后」的时序执行。先看全貌：

**核心文件清单**

| 文件 | 职责 |
|---|---|
| [`memoryTypes.ts`](https://github.com/binarylei/claudecode/blob/main/src/memdir/memoryTypes.ts) | 四种记忆类型定义、XML 模板、排除清单、召回验证指令 |

| [`findRelevantMemories.ts`](https://github.com/binarylei/claudecode/blob/main/src/memdir/findRelevantMemories.ts) | 查询时召回：Sonnet sideQuery 从所有 topic 中挑最相关的 ≤5 个 |
| [`extractMemories.ts`](https://github.com/binarylei/claudecode/blob/main/src/services/extractMemories/extractMemories.ts) | 后台提取：`hasMemoryWritesSince` 互斥、合并发送、追赶提取 |
| [`sessionMemory.ts`](https://github.com/binarylei/claudecode/blob/main/src/services/SessionMemory/sessionMemory.ts) | Session Memory 主逻辑：hook 注册、阈值触发、forked agent 执行 |
| [`sessionMemoryCompact.ts`](https://github.com/binarylei/claudecode/blob/main/src/services/compact/sessionMemoryCompact.ts) | Session Memory 替代 Compact：零成本摘要路径 |

**完整调用链**

```
启动时 [加载]
  └─ systemPromptSection('memory')  ← 加载 MEMORY.md 索引
       └─ loadMemoryPrompt() 

每轮 query
  ├─ autoCompactIfNeeded()
  │    └─ trySessionMemoryCompaction()  ← Session Memory 替代 Compact
  │
  ├─ startRelevantMemoryPrefetch()  ← 按需召回
  │    └─ sideQuery → ≤5 个 topic
  │
  ├─ callModel 循环               ← 路径A：模型消费 prompt 中的记忆指令
  │    └─ 模型返回 tool_use → 工具执行
  │         ├─ 可能：Write/Edit 到记忆目录  ──────────────────┐
  │         └─ 可能：无记忆写入                               │
  │                                                          │
  └─ executeExtractMemories()     ← 路径B：事后检查同一段消息 ─┘
       └─ hasMemoryWritesSince()  ← 扫描 assistant 消息的 tool_use block
            ├─ 已写入 → 跳过
            └─ 未写入 → 可能触发 fork Agent 提取
```

调用链值得关注两个设计惯例：**`autoCompactIfNeeded()` 在 `callModel` 之前执行**——为即将发出的请求腾空间，和 `startRelevantMemoryPrefetch()` 异步并行，不阻塞主流程。**路径 A 和路径 B 通过 `hasMemoryWritesSince()` 连接**——B 不关心模型"想什么"，只扫描本轮 tool_use block 的 `file_path` 是否落在记忆目录，已写入则跳过，避免重复 API 消费。

### 3.2 记忆的类型定义、写入规范与使用守则

[memoryTypes.ts](https://github.com/binarylei/claudecode/blob/main/src/memdir/memoryTypes.ts) 是记忆系统的「语法书」——它不执行任何逻辑，但定义了模型理解记忆的全部词汇：什么是记忆、怎么写、怎么用。以下按这条认知链展开。

#### 记忆的边界：四种类型 + 排除清单

四种类型（user / feedback / project / reference）以 XML 结构定义，每种包含 `name`、`description`、`when_to_save`、`how_to_use` 和 `examples`。

比类型定义更关键的是排除清单。`WHAT_NOT_TO_SAVE_SECTION` 在「不该记」上的措辞比「该记」更坚决——甚至连用户的显式请求都可以拒绝：

- 代码模式、约定、架构、文件路径 → 通过读代码推导
- Git 历史、最近变更 → `git log` / `git blame` 是权威源
- 调试方案或修复配方 → 修正在代码里，上下文在 commit message 里
- CLAUDE.md 已有的内容
- 临时任务细节：进行中的工作、临时状态、当前对话上下文

设计意图很清晰：类型定义是「建议」——语气可以柔和；排除清单是「禁令」——措辞必须绝对。因为「该记」靠判断，模型有容错空间；「不该记」靠禁令，模糊等于空文。

#### 写入规范：文件格式 + 两步保存法

`MEMORY_FILE_FORMAT_SECTION` 定义了 topic 文件的标准格式——frontmatter 声明 `name`（slug）、`description`（一句话摘要）、`metadata.type`（四种类型之一），正文承载具体内容。frontmatter 和正文分离的意图是让扫描时只需读元数据就能判断相关性，不用打开每个文件看全文。

格式模板中还有一个独特机制——`[[name]]` 跨文件链接。模型可以在一条记忆的正文中引用另一条记忆，格式与 MEMORY.md 索引行一致。这允许记忆之间形成关联网络，而非彼此孤立的碎片。

`MEMORY_SAVE_INSTRUCTIONS` 教模型分两步写入：先 Write topic 文件，再 Edit MEMORY.md 加一行指针。两个独立的 tool call，互不干扰。

#### 使用守则：验证优先于信任

`TRUSTING_RECALL_SECTION`（"Before recommending from memory"）告诉模型：记忆声称某个文件存在 → 先 grep 验证；记忆声称某个函数可用 → 先读源码确认。"The memory says X exists" is not the same as "X exists now."

### 3.3 System Prompt 中记忆指令的组装

memoryTypes.ts 定义了记忆的词汇，[memdir.ts](https://github.com/binarylei/claudecode/blob/main/src/memdir/memdir.ts) 负责把这些词汇组装成 prompt 注入 system prompt。

[`loadMemoryPrompt()`](https://github.com/binarylei/claudecode/blob/main/src/memdir/memdir.ts#L419-L507) 在启动时做了两件事：调用 [`buildMemoryLines()`](https://github.com/binarylei/claudecode/blob/main/src/memdir/memdir.ts#L199-L271) 将类型模板、排除清单、保存指令、验证指令拼装成「记忆使用手册」——这是静态的，每次会话相同；同时读取 MEMORY.md 索引全文追加到 prompt 末尾——这是动态的，随记忆增减变化。

当 MEMORY.md 索引文件条目超过 200 行，需要截断保护：

```typescript
// memdir.ts:57-103 — truncateEntrypointContent
export function truncateEntrypointContent(raw: string): EntrypointTruncation {
  // 先按行截断（自然边界）
  let truncated = wasLineTruncated
    ? contentLines.slice(0, MAX_ENTRYPOINT_LINES).join('\n')
    : trimmed

  // 再按字节截断（在最后一个完整换行处切断）
  if (truncated.length > MAX_ENTRYPOINT_BYTES) {
    const cutAt = truncated.lastIndexOf('\n', MAX_ENTRYPOINT_BYTES)
    truncated = truncated.slice(0, cutAt > 0 ? cutAt : MAX_ENTRYPOINT_BYTES)
  }

  // 附加精确的截断原因
  return {
    content: truncated + `\n\n> WARNING: ${ENTRYPOINT_NAME} is ${reason}...`,
    ...
  }
}
```

设计要点：

- **双重截断**：行数和字节数两个独立上界，分别应对「条目太多」和「单条太胖」。字节截断用 `lastIndexOf('\n')` 在完整换行处切断，避免截断一行中间的字符
- **显式告警**：截断后在内容末尾附加 WARNING，说明原因和规模——模型看到警告后会倾向于清理老旧条目

### 3.4 记忆写入兜底

3.2 的类型模板和 3.3 的 prompt 组装让模型在对话中主动写记忆（路径 A）——这是机会性的，触发与否取决于模型在对话中的判断，不保证每轮都发生。路径 B 用系统节奏覆盖 A 触及不到的窗口：每轮 query 结束后固定检查，扫描本轮 assistant 消息的 tool_use block，若未发现记忆写入且节流通过，则 fork Agent 提取。

#### 触发条件

每次 query loop 结束后，系统检查两个条件：一是 `turnsSinceLastExtraction` 计数达到间隔阈值（默认每 1 轮），二是主 Agent 在这轮对话中没写过记忆文件。后者通过 [`hasMemoryWritesSince()`](https://github.com/binarylei/claudecode/blob/main/src/services/extractMemories/extractMemories.ts#L121-L148) 判断——遍历 assistant 消息中的 tool_use block，检查是否有写入落在记忆目录内：

```typescript
// extractMemories.ts:121-148 — 互斥检查
function hasMemoryWritesSince(messages, sinceUuid): boolean {
  for (const message of messages) {
    if (message.type !== 'assistant') continue
    for (const block of message.message.content) {
      const filePath = getWrittenFilePath(block)
      if (filePath !== undefined && isAutoMemPath(filePath)) {
        return true  // 主 Agent 已写入 → 跳过
      }
    }
  }
  return false
}
```

检查是轻量的——只看 `file_path` 前缀是否落在记忆目录，不关心内容。主 Agent 已免费写过的内容，后台提取自动跳过，避免重复 API 消费。

#### 并发控制

如果上一轮提取还在执行中，新请求不丢弃，而是暂存为 `pendingContext`：

```typescript
// extractMemories.ts:557-564 — 合并暂存
if (inProgress) {
  pendingContext = { context, appendSystemMessage }
  return  // 立即返回，不阻塞主循环
}
```

当前提取的 `finally` 块中自动执行追赶——这一轮实际处理了两轮的积累对话，避免提取跟不上对话速度。

#### 成本控制

两次体现「能免费就不付费，付费也要省着花」：

**预算上限**：每次提取限制最多 5 轮 tool call——"读到 → 写入 → 停止"，不允许去 grep 验证记忆的真实性。防止 Agent 陷入验证循环，把免费的补漏变成昂贵的探索。

**缓存共享**：后台提取使用 `runForkedAgent`，fork 出的 Agent 复用主 Agent 的 system prompt 和对话历史前缀，使 prompt cache 命中率达到 98% 以上。路径 B 虽需额外的 API 调用，但大部分输入 token 走缓存，实际开销极低。

### 3.5 双层召回实现

[findRelevantMemories.ts](https://github.com/binarylei/claudecode/blob/main/src/memdir/findRelevantMemories.ts) 的入口函数展示了清晰的职责分离：

```typescript
// findRelevantMemories.ts:39-75
export async function findRelevantMemories(
  query: string, memoryDir: string, signal: AbortSignal,
  recentTools: readonly string[] = [],
  alreadySurfaced: ReadonlySet<string> = new Set(),
): Promise<RelevantMemory[]> {
  // 1. 扫描目录，过滤已出现的
  const memories = (await scanMemoryFiles(memoryDir, signal))
    .filter(m => !alreadySurfaced.has(m.filePath))

  // 2. Sonnet 挑选
  const selectedFilenames = await selectRelevantMemories(
    query, memories, signal, recentTools)

  // 3. 匹配回完整路径 + mtime
  return selected.map(m => ({ path: m.filePath, mtimeMs: m.mtimeMs }))
}
```

设计要点：

- **`alreadySurfaced`**：系统追踪已在此会话中注入过的记忆路径，避免 Sonnet 反复挑选同一条记忆浪费 5 槽预算
- **`recentTools`**：当 Agent 正在使用的工具恰好匹配某条记忆的主题时，不会选取该工具的「使用文档」——但保留「警告和陷阱」——因为正在使用时恰恰是最需要警告的时候

Sonnet 调用的关键是结构化输出：

```typescript
// findRelevantMemories.ts:98-122 — sideQuery with json_schema
const result = await sideQuery({
  model: getDefaultSonnetModel(),
  system: SELECT_MEMORIES_SYSTEM_PROMPT,
  messages: [{ role: 'user', content: `Query: ${query}\n\nAvailable memories:\n${manifest}` }],
  output_format: {
    type: 'json_schema',
    schema: {
      type: 'object',
      properties: { selected_memories: { type: 'array', items: { type: 'string' } } },
      required: ['selected_memories'],
    },
  },
})
```

使用 `json_schema` 而非自由文本返回，确保输出是精确的文件名列表——杜绝了模型「描述该选哪个文件」而非「返回文件名」的模糊输出。

[memoryScan.ts](https://github.com/binarylei/claudecode/blob/main/src/memdir/memoryScan.ts) 的 `scanMemoryFiles` 采用了单次扫描策略：`readFileInRange` 内部做 stat 获取 mtime，所以一次读取同时拿到元数据和内容——不需要先 stat 排序再读。在 N ≤ 200 的常见情况下，这比两轮系统调用少一半开销。

召回的最终注入点是在 [attachments.ts](https://github.com/binarylei/claudecode/blob/main/src/utils/attachments.ts#L2215-L2241) 中，以 `relevant_memories` 类型 attachment 的形式出现。每条记忆的全文内容前附加时效性提示（[memoryAge.ts](https://github.com/binarylei/claudecode/blob/main/src/memdir/memoryAge.ts)）——"today"、"yesterday"或"N days ago"，超过 1 天则附带 `<system-reminder>` 验证警告。系统还追踪每个会话内已注入的总字节数，防止记忆注入消耗过多上下文。

### 3.6 Session Memory 实现

[Session Memory](https://github.com/binarylei/claudecode/blob/main/src/services/SessionMemory/sessionMemory.ts) 的实现展示了「后台异步、最小打扰」的设计哲学。

注册发生在启动时（[sessionMemory.ts:357-375](https://github.com/binarylei/claudecode/blob/main/src/services/SessionMemory/sessionMemory.ts#L357-L375)），但 gate check 和 config 加载都是懒加载的——注册 hook 时不做任何判断，直到 hook 实际触发时才检查。这样做的好处是避免了启动时阻塞。

每次触发时的执行流程：

```typescript
// sessionMemory.ts:272-350 — extractSessionMemory hook
const extractSessionMemory = sequential(async function (context) {
  // 1. 只在主线程运行
  if (querySource !== 'repl_main_thread') return

  // 2. 懒加载 gate check
  if (!isSessionMemoryGateEnabled()) return

  // 3. 懒加载 config
  initSessionMemoryConfigIfNeeded()

  // 4. 阈值判断
  if (!shouldExtractMemory(messages)) return

  // 5. 设置文件，读取当前状态
  const { memoryPath, currentMemory } = await setupSessionMemoryFile(setupContext)

  // 6. 构建 prompt + 执行 forked agent
  await runForkedAgent({
    promptMessages: [createUserMessage({ content: userPrompt })],
    cacheSafeParams: createCacheSafeParams(context),
    canUseTool: createMemoryFileCanUseTool(memoryPath),
    querySource: 'session_memory',
    forkLabel: 'session_memory',
  })
})
```

设计要点：

- **`sequential()` 包装**：确保同一时刻只有一个提取在运行——Session Memory 更新必须串行，否则两个并发的 Edit 会产生冲突
- **工具极简限制**：`createMemoryFileCanUseTool()` 只允许 FileEditTool 操作**唯一**的 session memory 文件路径——Agent 不能读其他文件、不能 grep、不能执行 bash。这是最严格的工具限制之一，因为 Session Memory Agent 的唯一任务就是更新笔记

更新 prompt（[prompts.ts:43-81](https://github.com/binarylei/claudecode/blob/main/src/services/SessionMemory/prompts.ts#L43-L81)）的核心约束是**结构保护**：

- 所有 `# Section Title` 行不可修改
- 所有斜体 `_description_` 行不可修改（那是模板说明，不是内容）
- 只允许更新每个 section 的实际内容
- 每个 section 预算 ≤2000 tokens，总文件 ≤12000 tokens

这些约束确保模板结构不被破坏——模板是 Session Memory 能用于 Compact 的前提。如果结构变形，Compact 的「用会话摘要替代 LLM 摘要」路径就无法可靠工作。Session Memory 是「加分项」而非「硬依赖」。

## 4. 总结

1. **只记不可推导**：代码结构、git 历史、CLAUDE.md 已覆盖的信息不存——记忆空间留给工具查不到的信息。四种类型的划分（user/feedback/project/reference）为这一边界提供了操作化的分类框架。

2. **能免费就不付费，付费也要省着花**：主 Agent 在对话中顺带写入记忆，增量成本为零。后台提取只在主 Agent 遗漏时补位，且通过互斥避免重复消费。Forked Agent 的缓存共享（98%+ 命中率）让「付费」路径的实际开销降到最低。

3. **全量索引 + 按需正文 = 双层召回**：MEMORY.md 全量注入 system prompt 提供全景认知（零成本），Sonnet sideQuery 按 query 挑选最相关的 ≤5 个 topic 文件注入上下文（1 次轻量 API）。宁可漏报不要误报——无关记忆挤占上下文窗口的代价比少召回更大。

4. **记忆是快照，不是真相**：人性化时间呈现（"47 days ago"）+ 过期 `<system-reminder>` 警告 + "Before recommending from memory" 验证指令——三层设计防止模型被记忆的「权威外观」误导，强制「验证优先于信任」。

5. **Session Memory 是零成本的 Compact 替代**：实时会话摘要异步维护，增量成本为零。Compact 触发时优先复用笔记作为摘要——省下一次 LLM 调用。这是上下文管理 Part 6 贯穿始终的经济逻辑在记忆系统中的收束。

6. **文件系统上的记忆可编辑、可追踪、可共享**：每条记忆是独立的 markdown 文件，人类可手动编辑删除，可纳入 git 版本控制，可设置 team 子目录实现团队共享——记忆不是黑盒，是透明的知识资产。
