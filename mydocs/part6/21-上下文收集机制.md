# 21. 上下文收集机制

本章是 Part 6（上下文管理）的开篇。在[第 6 章](../part2/06-核心循环-下-Agent-Loop迭代机制)中我们建立了 Agent Loop 的迭代模型——每轮循环的第一步就是"组装上下文、调用 API"。本章回答这个"第一步"的具体机制：**Claude Code 如何将散落在各处的信息——系统指令、用户配置、对话历史、环境状态、动态附件——组装成一条完整的 LLM API 请求？**

全文围绕 **WHAT（收集什么）→ HOW（如何组装）→ WHEN（何时收集）** 三个维度展开，核心线索是**"不同变化频率的内容走不同的注入通道和缓存策略"**。

## 1. 背景介绍

### 1.1 什么是上下文收集

> 上下文收集机制是 Claude Code 的"记忆输入管线"——在每个 Agent Loop 轮次开始前，将系统提示词、用户配置文件（CLAUDE.md）、对话历史、环境信息、动态附件等按优先级和缓存策略组装为一条完整的 LLM 输入。

理解这个定义的最佳方式，是将 Claude Code 与一个简单的 chatbot 做对比：

```
简单 chatbot（固定拼装）                Claude Code（动态管线）

  system: "You are a helpful AI"         system: 静态前缀（global cache）
  + 最近 20 条消息                          + 动态后缀（org cache）
  → 直接拼接，无缓存策略                     + CLAUDE.md 注入（userContext）
                                            + git status（systemContext）
                                            + 对话历史（JSONL 恢复链路）
                                            + 动态附件（delta 增量注入）
                                            → 6 层组装 + 3 级缓存分区
```

关键差异一句话说清：**简单系统只有"拼接"，Claude Code 有"组装管线"——变化频率不同的内容走不同的注入通道，用不同的缓存策略。** 这套管线不是"把所有东西拼成一个字符串"的体力活，而是一个精心设计的**分层架构**——每一层的设计决策直接影响 prompt cache 命中率和 token 成本。

### 1.2 为什么上下文收集值得单独一章

在第 5-7 章中，我们已经看到 Agent Loop、Token 管理、上下文收集三者的关系：

```
Agent Loop 每轮迭代
  │
  ├─► ① 上下文收集 ◄── 本章主角
  │      ├─ System Prompt 组装
  │      ├─ CLAUDE.md 注入
  │      ├─ 对话历史加载
  │      └─ 动态附件收集
  │
  ├─► ② callModel（发送到 API）
  │      └─ Token 计数与预算控制（第 7 章）
  │
  └─► ③ 工具执行 → 回流 → 下一轮
```

上下文收集决定了 API 调用的**输入质量**和**缓存效率**。一个设计不当的收集策略会导致：
- **缓存命中率低**：高频变化的内容混入低频变化的段，导致整个 prompt cache 频繁失效
- **Token 浪费**：重复发送不变的工具列表、Agent 列表
- **上下文膨胀**：无关的 CLAUDE.md 文件被注入

这引出了本章的核心问题：**如何在信息完整性和缓存效率之间取得平衡？**

### 1.3 正交分解：三个子问题

将上下文收集机制分解为三个独立维度：

```
                     上下文收集机制

       WHAT                        HOW                        WHEN
  ┌──────────────┐          ┌──────────────┐          ┌──────────────┐
  │ 收集什么？     │          │ 如何组装？     │          │ 何时收集？     │
  │              │          │              │          │              │
  │ · 系统提示词  │          │ · 分层拼接    │          │ · 启动预加载   │
  │ · 用户配置    │          │ · 优先级选择  │          │ · 每轮动态收集 │
  │ · 对话历史    │          │ · 缓存分区    │          │ · Delta 延迟  │
  │ · 环境状态    │          │ · 增量注入    │          │ · 异步预取    │
  │ · 动态附件    │          │              │          │              │
  └──────────────┘          └──────────────┘          └──────────────┘
```

WHAT 是数据来源维度——上下文从哪些渠道收集；HOW 是架构维度——如何将这些来源组装成结构化的请求；WHEN 是时序维度——什么时机触发收集。三个维度互相独立又环环相扣：不同来源的数据有不同的变化频率，不同的变化频率决定了不同的缓存策略和收集时机。

## 2. 核心逻辑

### 2.1 变化频率决定缓存策略：从字符串到三层缓存分区

上下文收集机制中最重要的设计决策不是"收集什么"，而是"把什么放在哪个缓存分区"。这个决策的唯一输入变量是**内容的变化频率**——变化越慢，缓存范围越大（global > org > 不缓存）。这条原则在三个子系统中独立得出相同结论，是整条管线的底层逻辑。

**分层组装模型**

在深入论证之前，先用一张图建立全局认知：

```
┌──────────────────────────────────────────────────┐
│              最终发送给 API 的上下文               │
├──────────────────────────────────────────────────┤
│                                                  │
│  ┌─ System Prompt（静态前缀）───────────────┐     │
│  │  identity / tone / tools / output        │     │  ← global cache
│  └──────────────────────────────────────────┘     │    （版本升级才变）
│                                                  │
│  ─── SYSTEM_PROMPT_DYNAMIC_BOUNDARY ───────     │  ← 缓存分界线
│                                                  │
│  ┌─ System Prompt（动态后缀）───────────────┐     │
│  │  memory / env / language / MCP           │     │  ← org cache
│  └──────────────────────────────────────────┘     │    （会话内可能变）
│                                                  │
│  ┌─ User Context ──────────────────────────┐     │
│  │  CLAUDE.md + currentDate                │     │  ← 不缓存
│  └──────────────────────────────────────────┘     │    （用户随时修改）
│                                                  │
│  ┌─ Messages + Attachments ────────────────┐     │
│  │  对话历史 + 动态附件（delta 增量）        │     │  ← 不缓存
│  └──────────────────────────────────────────┘     │
│                                                  │
└──────────────────────────────────────────────────┘
```

这张图重点不在各层内部装了什么，而在**四层之间的三个分隔边界**——每个边界对应一种缓存策略的切换。边界画在哪里，直接决定 prompt cache 命中率和 API 成本。

**方案演进：从字符串到缓存分界线**

理解这个设计为什么长这样，最好的方式是看它经历了怎样的演进：

**v1：一个固定字符串。** 把系统提示词写死成一个字符串常量。问题：CLI 主 Agent、IDE Bridge、子 Agent、Workflow 协调器四种场景需求完全不同，一个固定字符串无法适配。而且 Anthropic API 的 prompt cache 以整个 prompt 的 hash 为 key，任何微小变化都导致缓存失效——固定的长字符串反而最浪费缓存。

> **举例**
>
> `"你是 Claude Code。CWD: /home/alice/foo"`
> 
> `"你是 Claude Code。CWD: /home/bob/bar"`
>
> 两条 prompt 仅 CWD 不同，但 API 按完整 hash 匹配缓存——视为两个独立 key。

**v2：分段拼接，按模式 switch。** 将 system prompt 按功能拆分为多个 section，根据 CLI / Bridge / Subagent 模式按需组合。代码组织更清晰，但缓存效率与 v1 一样糟糕——每次调用重新拼接，hash 值每次都不同。

> **举例**
>
> `拼装结果 A: [identity] + [tools] + "Alice 偏好 X"`
> 
> `拼装结果 B: [identity] + [tools] + "Bob 偏好 Y"`
>
> 动态段一变，整条 prompt hash 全变，静态段跟着遭殃。

**v3（最终方案）：静态前缀 + 动态边界 + memoized sections。** 核心思想：在 system prompt 中画一条"缓存分界线"。分界线之前的内容（identity、tools、tone）只在版本升级时变化 → global cache。分界线之后的内容（memory、env、language）含用户/会话特定信息 → org cache。最关键的信息在最显式的位置——一条名为 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 的常量，它本身不贡献语义，纯粹是给下游的 `splitSysPromptPrefix()` 做切分标记。这是**用约定替代配置**的典型案例——不需要在函数签名中增加额外参数，用数据本身的特殊值标记结构边界。

> **举例**
>
> `[identity + tools + tone]`  ← global cache（所有用户共享）
> 
> `────── boundary ──────`
> 
> `[memory + env + language]`  ← org cache（按组织隔离）
>
> 用户 A 换成 B，边界之上不变 → global cache 命中；边界之下按组织隔离 → 互不干扰。

实际实现中，7 个动态 section 有 6 个采用 memoized 策略——缓存是默认，破坏缓存是例外。这也印证了 v3 方案的核心判断："动态性"是少数情况，而非普遍规则。

**跨子系统的一致性**

"变化频率决定缓存策略"这条原则不仅体现在 System Prompt 内部。它在另外两个子系统中独立得出相同结论：

**CLAUDE.md 注入通道。** CLAUDE.md 的内容不走 system prompt，而是走 `userContext` 通道。原因：用户随时可能修改配置文件。如果混入 system prompt，每次改一个字都导致整个 prompt cache 失效。这是以"数据归属"换"缓存效率"的决策——CLAUDE.md 本质上是用户数据而非系统指令，它不应该参与系统指令的缓存 key 计算。

**Delta 附件。** 工具列表、Agent 列表、MCP 指令在绝大多数轮次中不变。delta 机制让这些内容只在首轮全量发送，后续只发变化部分。token 消耗从 O(n × 轮次) 降到 O(n + 变化量 × 轮次)。更精妙的是，当 delta 机制启用时，MCP 指令甚至完全绕过 section 系统——`DANGEROUS_uncached` 的 compute 函数直接返回 `null`，从架构层面消除了缓存破坏问题。这是"用架构优化消除实现层面痛苦"的典范。

### 2.2 优先级责任链：同一管线适配四场景

上下文收集管线需要服务 CLI 主 Agent、IDE Bridge、子 Agent、Workflow 协调器四种场景。不同场景对 system prompt 的需求截然不同——CLI Agent 需要交互指令，Bridge Agent 需要 IDE 通信协议，子 Agent 需要领域知识，协调器需要编排逻辑。

最简单的做法是用 switch-case 按场景硬编码分支。但 4 种场景 × 3 种关系（替换/追加/透传）会产生指数级分支，每增加一种模式所有分支都要重新考虑。Claude Code 选择了**责任链模式**——每层只需决定"我对下一层做什么"，不关心上层做了什么决定：

```
优先级从高到低：

┌─────────────────────────────────────────────────┐
│ Override（--system-prompt）                       │
│ 关系：完全替换所有层级                              │
│ 意图：用户最终控制权——"我给你一票否决权"              │
├─────────────────────────────────────────────────┤
│ Coordinator（多 Agent 协调器模式）                  │
│ 关系：替换 default，保留 append                    │
│ 意图：多 Agent 编排需要专用系统指令                  │
├─────────────────────────────────────────────────┤
│ Agent Definition（子 Agent 定义文件）               │
│ 关系：Proactive → 追加 / 其他 → 替换               │
│ 意图：Agent 定义是"领域知识叠加"而非"系统替代"        │
├─────────────────────────────────────────────────┤
│ Custom / Default（用户配置 / 内置默认）              │
│ 关系：custom 未设置时 fallback 到 default           │
│ 意图：提供开箱即用的默认值                          │
├─────────────────────────────────────────────────┤
│ Append Prompt                                    │
│ 关系：所有路径都追加                               │
│ 意图：全局指令（合规声明、调试信息）                  │
└─────────────────────────────────────────────────┘
```

**为什么责任链优于 switch-case。** 责任链让每层独立封装一种"关系决策"——替换、追加还是透传。如果 Override 被设置，Coordinator 层根本不会执行到。如果 Coordinator 被激活，Agent 层的决策被跳过。这种"短路"语义在责任链中是天然的，在 switch-case 中却需要额外的条件判断。

**Proactive 模式为什么选择"追加"。** 这是优先级链中最关键的决策。自主 Agent 仍然需要默认 prompt 中的环境信息（操作系统、shell 类型）、安全约束（权限规则）和工具使用规则。Agent 定义文件提供的是领域知识——"你是一个代码审查专家"——这些应该叠加在基础能力之上。如果 Agent 定义替换了 default，自主 Agent 就失去了环境感知和安全边界。

### 2.3 增量追加与惰性计算：只传输变化的部分

上下文管线的效率不仅取决于"缓存什么"，还取决于"重算什么"和"重传什么"。Claude Code 在三个子系统中独立实施了同一策略——**默认全量，按需增量；默认惰性，按需刷新**：

| 子系统 | 全量方案 | 问题 | 增量/惰性方案 | 效果 |
|--------|---------|------|--------------|------|
| 对话历史 | 全量覆盖 JSON 文件 | 写入中途崩溃 → 整个会话损坏 | JSONL 增量追加 | 最多丢失最后一条消息 |
| System Prompt | 每轮重算所有 section | 浪费计算，微小变化破坏缓存 | 6/7 memoized 惰性计算 | 只在 /clear、/compact 时刷新 |
| 动态附件 | 每轮全量重发工具列表 | O(n × 轮次) token | Delta 增量 | O(n + 变化量 × 轮次) |

**JSONL 增量追加。** 对话以 JSONL 格式持久化，写入采用三步增量模型：

1. **去重**：将传入消息的 UUID 与已写入会话的 UUID Set 对比，过滤已存在的消息。O(1) 的 Set 查找保证在数千条消息的会话中性能不退化
2. **追加**：将新消息追加写入 JSONL 文件末尾，而非覆盖整个文件
3. **返回锚点 UUID**：返回最后一个链参与者的 UUID，供下一批消息建立正确的 parent 链接

核心权衡是增量追加 vs 全量覆盖：增量追加的代价是需要维护 UUID 去重逻辑和 parent 链，但换来崩溃恢复能力——进程在写入中途被 kill 时，最多丢失最后一条消息，整个会话文件不受影响。

**parentUuid 链重建。** JSONL 文件的物理顺序不等于对话的逻辑顺序——尤其是在 compaction 后，旧消息被 summary 替换，新消息追加在末尾：

```
物理 JSONL 顺序：         逻辑对话链（通过 parentUuid 重建）：

  msg-1 (parent: null)      msg-1 ──→ msg-2 ──→ msg-3 ──→ msg-4
  msg-3 (parent: msg-2)               ↑
  msg-2 (parent: msg-1)         compact summary
  compact-summary                 (替代 msg-1,2,3)
  msg-4 (parent: compact)
```

这种设计将"物理存储顺序"和"逻辑对话顺序"解耦——compaction 可以安全地删除旧消息，只要 `msg-4` 的 `parentUuid` 指向 `compact-summary`，重建时就能正确还原对话链。

**Delta 附件。** 对于工具列表、Agent 列表、MCP 指令这类"绝大多数轮次中不变"的信息，delta 机制让系统只在首轮全量发送，后续只发送变化的部分。如果 40+ 个工具的描述每轮都重发，单是工具定义就可能占数千 token。Delta 让这部分的 token 消耗从线性增长降为"仅在变化时增长"。

**两处惰性计算、一处增量传输，同一条原则。** 三个子系统面对的问题不同——持久化可靠性、计算开销、传输冗余——但解决方案共享同一个核心洞见：**识别出"不变"的部分，用更少的代价处理它们。** Memoized section 避免重算不变的内容，JSONL 追加避免重写不变的记录，Delta 附件避免重传不变的列表。

### 3.2 完整调用链路

上下文收集存在两条独立调用路径：**CLI 交互路径**（终端用户输入，本章主体）和 **SDK 编程路径**（`QueryEngine.ask()`，程序化调用）。两条路径在 `query()` 汇合，共享下游的缓存分区和 API 调用逻辑。

下面的链路以 CLI 路径为主线，SDK 路径的差异在末尾单独说明。

**关键时序**：`queryLoop` 每轮迭代分为两个阶段——先组装请求并调用 API（阶段一），再执行工具并收集系统附件（阶段二）。系统附件（delta、changed files 等）在**本轮 API 响应后才注入**，通过 `state.messages` 带入下一轮，模型在**第二轮**才能看到。用户触发的附件（`@文件`、MCP 资源等）则在进入 `query()` **之前**已由 REPL 层处理完毕，直接通过 `messages` 参数传入。

```
REPL.tsx（用户输入触发，每轮 Agent Loop 迭代）
  │
  ├─► Promise.all([                              // 并行获取上下文三件套
  │     getSystemPrompt(tools, model, …),        // constants/prompts.ts
  │     │     │
  │     │     │  // 三种复杂度分支
  │     │     ├─► CLAUDE_CODE_SIMPLE? → 极简 prompt（1 个字符串）
  │     │     ├─► Proactive 激活?  → 精简自主 Agent prompt
  │     │     └─► Standard 路径 ↓
  │     │           ├─► [静态段数组]  // 硬编码 PromptText[]
```

### 3.4 静态/动态分界：getSystemPrompt

`getSystemPrompt()` 的核心结构展示了静态段和动态段如何协同工作。

```typescript
// constants/prompts.ts:444-549
export async function getSystemPrompt(
  tools: Tools, model: string,
  additionalWorkingDirectories?: string[],
  mcpClients?: MCPServerConnection[],
): Promise<string[]> {
  // 提前返回：CLAUDE_CODE_SIMPLE 模式
  if (isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE)) {
    return [
      `You are Claude Code, Anthropic's official CLI for Claude.\n\n` +
      `CWD: ${getCwd()}\nDate: ${getSessionStartDate()}`,
    ]
  }

  // Proactive 模式：精简自主 Agent prompt
  if (proactiveModule?.isProactiveActive()) {
    return [
      `\nYou are an autonomous agent. ...`,
      getSystemRemindersSection(),
      await loadMemoryPrompt(),
      envInfo,
      getLanguageSection(settings.language),
      // ... 精简段列表
    ].filter(s => s !== null)
  }

  // 标准路径：静态段 + 动态边界 + memoized 动态段
  const dynamicSections = [
    systemPromptSection('session_guidance', () => getSessionSpecificGuidanceSection(...)),
    systemPromptSection('memory', () => loadMemoryPrompt()),
    // ...
    DANGEROUS_uncachedSystemPromptSection('mcp_instructions', () => ..., 'MCP servers ...'),
    // ...
  ]

  return [
    // 静态段（硬编码）
    getIdentitySection(model),
    getSystemRemindersSection(),
    getHooksSection(),
    getToneAndStyleSection(tools),
    getDoingTasksSection(),
    getActionsSection(),
    getUsingToolsSection(),
    getOutputEfficiencySection(),
    // 缓存分界线
    SYSTEM_PROMPT_DYNAMIC_BOUNDARY,
    // 动态段（memoized）
    ...(await resolveSystemPromptSections(dynamicSections)),
  ].filter(s => s !== null)
}
```

设计要点：

- **三种复杂度层级**：SIMPLE（极简，1 个字符串）→ Proactive（精简，自主体）→ Standard（完整，交互式）。每种路径的 system prompt 体量差异巨大，但共用同一个函数入口
- **`SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 是一个占位字符串**：它本身不贡献语义内容，纯粹是给下游的 `splitSysPromptPrefix()` 做"切分标记"。这是"数据中嵌入元数据"的模式——用数据本身的特殊值标记结构边界，避免在函数签名中增加额外参数

> **延伸阅读**：[附录1：系统级上下文组成](../附录/附录1-系统级上下文组成) 完整列出模型接收到的系统级上下文——System Prompt（7 静态 + 13 动态段）+ systemContext（环境快照）+ userContext（项目个性化）的结构、内容与设计要点。

### 3.5 CLAUDE.md 加载：getUserContext

getUserContext 方法从 CWD 向上遍历目录树收集配置文件 CLAUDE.md ，经 `@include` 展开和格式化后注入 `userContext`。它不走 system prompt 是为了保护 prompt cache 不受用户频繁修改配置文件的冲击。`getMemoryFiles()` 按以下优先级收集配置文件：src/claudemd.ts:790

1. **Managed** — `/etc/claude-code/CLAUDE.md`（系统管理员统一配置）
2. **User** — `~/.claude/CLAUDE.md`（跨项目个人偏好）
3. **Project** — 从 CWD 向上遍历到根目录，在每级收集 `CLAUDE.md`、`.claude/CLAUDE.md`、`.claude/rules/*.md`
4. **Local** — `CLAUDE.local.md`（个人私有，不检入版本控制）
5. **AutoMem / TeamMem** — 自动记忆与团队记忆入口

### 3.6 缓存分区：splitSysPromptPrefix

`splitSysPromptPrefix()` 将扁平的 system prompt 数组转换为带缓存标记的结构化块。

```typescript
// utils/api.ts:321-381
export function splitSysPromptPrefix(
  systemPrompt: SystemPrompt,
  options?: { skipGlobalCacheForSystemPrompt?: boolean },
): SystemPromptBlock[] {
  const useGlobalCacheFeature = shouldUseGlobalCacheScope()

  // 路径 1：MCP 工具存在 → 降级为 org 缓存
  if (useGlobalCacheFeature && options?.skipGlobalCacheForSystemPrompt) {
    let attributionHeader, systemPromptPrefix
    const rest: string[] = []
    for (const prompt of systemPrompt) {
      if (prompt === SYSTEM_PROMPT_DYNAMIC_BOUNDARY) continue
      if (prompt.startsWith('x-anthropic-billing-header'))
        attributionHeader = prompt
      else if (CLI_SYSPROMPT_PREFIXES.has(prompt))
        systemPromptPrefix = prompt
      else rest.push(prompt)
    }
    // → attributionHeader: cacheScope=null
    // → systemPromptPrefix: cacheScope='org'
    // → rest: cacheScope='org'
  }

  // 路径 2：Global cache + 找到动态边界 → 最优路径
  if (useGlobalCacheFeature) {
    const boundaryIndex = systemPrompt.findIndex(
      s => s === SYSTEM_PROMPT_DYNAMIC_BOUNDARY)
    if (boundaryIndex !== -1) {
      // 边界之前 → cacheScope='global'
      // 边界之后 → cacheScope=null
    }
  }

  // 路径 3：默认路径（3P providers 或无边界） → org 缓存
  // → 整个 system prompt → cacheScope='org'
}
```

设计要点：

- **三种缓存策略的降级逻辑**：最优路径是"global + boundary found"，最差路径是"org only"。降级条件不是错误——它们是架构对"当前环境不支持最优策略"的适应性反应
- **MCP 工具存在时降级为 org 缓存的原因**：MCP 工具的 schema 可能跨组织不同（不同组织的 MCP 服务器配置不同），使用 global cache 可能读到其他组织的缓存。这是一个"安全性优先于性能"的决策
- **attribution header 始终 `cacheScope=null`**：因为 billing header 的平台前缀因部署而异，需要始终从缓存中排除

### 3.7 对话持久化：recordTranscript 的增量写入

`recordTranscript()` 展示了增量写入和去重机制（[sessionStorage.ts:1408-1448](https://github.com/binarylei/claudecode/blob/main/src/utils/sessionStorage.ts#L1408-L1448)）：

```typescript
export async function recordTranscript(
  messages: Message[],
  teamInfo?: TeamInfo,
  startingParentUuidHint?: UUID,
  allMessages?: readonly Message[],
): Promise<UUID | null> {
  const cleanedMessages = cleanMessagesForLogging(messages, allMessages)
  const sessionId = getSessionId() as UUID
  const messageSet = await getSessionMessages(sessionId)  // UUID Set
  const newMessages: typeof cleanedMessages = []
  let startingParentUuid: UUID | undefined = startingParentUuidHint
  let seenNewMessage = false

  for (const m of cleanedMessages) {
    if (messageSet.has(m.uuid as UUID)) {
      // 已存在 → 跳过，但更新 parent 指针
      if (!seenNewMessage && isChainParticipant(m)) {
        startingParentUuid = m.uuid as UUID
      }
    } else {
      newMessages.push(m)
      seenNewMessage = true
    }
  }

  if (newMessages.length > 0) {
    await getProject().insertMessageChain(
      newMessages, false, undefined, startingParentUuid, teamInfo,
    )
  }

  const lastRecorded = newMessages.findLast(isChainParticipant)
  return lastRecorded?.uuid ?? startingParentUuid ?? null
}
```

设计要点：

- **UUID Set 去重的复杂度是 O(1)**：`getSessionMessages()` 返回一个 Set 而不是 Array——对于可能有数千条消息的会话，O(1) vs O(n) 的差异显著
- **`seenNewMessage` 标志处理前缀跳过**：compaction 后，旧消息被 summary 替代，但它们的 UUID 仍在 `messageSet` 中。`seenNewMessage` 确保这些旧消息被安全跳过而不干扰新的 parent 链
- **`isChainParticipant` 不是所有消息**：progress 消息（工具执行的中间状态）写入 JSONL 但不形成对话链——它们只是日志，不应该被模型"记住"


### 3.8 设计要点汇总

本节将各子系统的设计要点按主题汇总：

**System Prompt 组装**

1. `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 将一条 prompt 分成两个缓存域——不是技术 hack，而是将"变化频率"显式化为架构边界
2. memoized section 注册表让"缓存什么、刷新什么"从隐式的代码逻辑变成显式的数据结构——`systemPromptSection` vs `DANGEROUS_uncachedSystemPromptSection` 的名字本身就是决策记录
3. `_reason` 参数在运行时不起作用，但它强制开发者在代码中留下设计意图——这是"可维护性优先于简洁性"的实践

**优先级选择**

4. 优先级链不是 if-else 分支，而是责任链模式——每一层可以替换、追加或透传。Proactive 模式的"追加"决策体现了对默认 prompt 中环境信息价值的尊重
5. Override 是"逃生舱"——系统给用户最终控制权，不管内部逻辑多复杂

**文件注入**

6. CLAUDE.md 走 `userContext` 而非 system prompt 通道——核心原因是保护 prompt cache 不受用户配置变化的冲击
7. `@include` 指令的 5 层深度上限是"够用且不失控"的平衡——既能支持模块化组织，又能防止无限膨胀
8. 文件加载优先级（Managed < User < Project < Local < AutoMem）体现了一个原则：越靠近当前任务上下文的配置，权重越高
9. `setCachedClaudeMdContent` 通过 state 中转 CLAUDE.md 内容供 auto-mode classifier 消费——避免 classifier 直接 import `claudemd.ts` 造成循环依赖，是解耦的常见手法

**对话历史**

10. JSONL 增量追加保证崩溃恢复能力——最多丢失最后一条消息而非整个会话
11. `parentUuid` 链重建将"物理存储顺序"和"逻辑对话顺序"解耦——compaction 可以安全地删除旧消息而不破坏后续消息的链接
12. 会话恢复不只是加载消息文本——还从 JSONL 中恢复 readFileState、attributionState、compact boundary、Agent/model override 和 worktree 状态，保证恢复后的 Agent 行为与中断前一致

**动态附件**

13. Delta 机制将工具/Agent/MCP 列表的 token 消耗从 O(n × 轮次) 降到 O(n + 变化量 × 轮次)
14. 附件收集的时序约束（用户触发先于系统触发）看似微妙，实则是正确性的保证——系统附件依赖用户附件填充的 trigger 数据
15. MCP 指令在 delta 机制启用后从 section 退化到 attachment——`DANGEROUS_uncached` 的 compute 直接返回 `null`，从架构层面消除了缓存破坏问题

**缓存分区**

16. `splitSysPromptPrefix` 的三条路径是缓存策略的"优雅降级"——不是错误处理，而是架构对不同环境的适应性反应
17. MCP 工具存在时跳过 global cache 是"安全优先于性能"的选择——global cache 可能跨组织共享，但 MCP 工具的 schema 因组织而异

## 4. 总结

1. **上下文收集是"分层组装 + 缓存分区 + 增量注入"的三位一体设计**——每一层有独立的变化频率、缓存策略和收集时机，不是简单的字符串拼接
2. **`SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 是全文最核心的设计决策**——它在一条 prompt 中画出缓存分界线，让高频变化段不影响低频变化段的缓存命中率，直接决定了 API 成本
3. **Memoized section 注册表将"缓存策略"从隐式逻辑变为显式声明**——`systemPromptSection` 和 `DANGEROUS_uncachedSystemPromptSection` 的命名本身就是给开发者的文档
4. **优先级链（override → coordinator → agent → custom → default）让同一套组装管线适配四种场景**——CLI 主 Agent、IDE Bridge、子 Agent、Workflow 协调器
5. **CLAUDE.md 走 `userContext` 通道而非 system prompt 是成本驱动的架构决策**——保护 prompt cache 不受用户频繁修改配置文件的冲击
6. **Delta 附件机制是"默认全量 → 按需增量"优化思路的典型案例**——工具列表不变就不重发，只在变化时发送增量
7. **对话历史的 JSONL 增量追加 + UUID 去重 + parentUuid 链重建**，保证崩溃恢复能力、去重正确性和压缩安全性
8. **`splitSysPromptPrefix` 的三条缓存路径不是错误处理，而是架构对环境约束的优雅降级**——从 global → org → 不缓存逐级 fallback

**覆盖边界**：本章聚焦上下文"收集与组装"阶段。上下文窗口已满时如何压缩（三层压缩策略）见第 22-23 章；记忆文件的检索、创建和更新机制见第 24 章。对话历史在 API 调用前的裁剪（micro-compact）见第 22 章。
